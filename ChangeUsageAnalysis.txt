Change Usage Analysis
=====================

Goal
----

`change` in Rocq proofs is a smell: it often means a lemma statement,
normalizer, notation, or regular fact does not line up with the proof goal.
This file tracks every class of `change` in `Denotation` and Soundness proofs,
then records cleanup iterations.

Scan commands
-------------

```sh
rg -n "\bchange\b" Denotation ContextTyping/theories/Soundness*.v -g '*.v'
rg -n "\bchange\b" Denotation ContextTyping/theories/Soundness*.v -g '*.v' \
  | cut -d: -f1 | sort | uniq -c | sort -nr
```

Classes
-------

1. Open/result-first normalization mismatch

- Examples:
  - `TypeEquivArrowResultFirst.v` and `TypeEquivTLet.v` had local
    `Hopened_scope` facts stated as raw
    `formula_open 0 f (FImpl (expr_result ...) (arrow_value ...))`.
  - Similar remaining patterns exist in `TypeEquivWandResultFirst.v`,
    TLet's Wand/Persist branches, and persistence Wand proof files.
- Cause:
  - The outer result slot is opened, then later normalized. The assertion
    itself carries the unnormalized form.
- Fix pattern:
  - Normalize the assertion statement at the source:
    opened result graph becomes `expr_result_formula_at (...) e (LVFree f)`;
    value body becomes `*_value_denote_gas_with ... (<[f := ...]> Σrel) ...
    (tret (vfvar f))`.

2. Opened result type is the actual subject

- Examples:
  - `Hres_src_regular` / `Hres_tgt_regular` with `cty_open 0 y τr`.
- Cause:
  - This is not just residue: the return type genuinely depends on the
    argument variable. These should not be blindly removed.
- Fix pattern:
  - Only normalize env/term opens around them. Keep the opened type unless a
    named opened-body abbreviation improves readability.

3. Store/resource representation coercions

- Examples:
  - many `change (a ∈ dom (σ : gmap atom value))`;
  - `change (dom (store_restrict σ X : StoreT) = X)`;
  - `change ((mfib : WorldT) = raw_fiber ...)`.
- Cause:
  - Store/World wrappers hide the underlying stdpp map/set operation from
    stdpp solvers.
- Fix pattern:
  - Move coercion-normalization lemmas/tactics to the lowest store/resource
    facts layer. Avoid repeated local `change`.

4. Erasure / notation folds

- Examples:
  - `change (erase_ty τx →ₜ erase_ty τ) with (erase_ty (CTArrow τx τ))`;
  - `change (erase_ty (CTPersist τ)) with (erase_ty τ)`.
- Cause:
  - The proof goal expects a folded erased type, while computation exposes the
    erased components.
- Fix pattern:
  - Add or use small erasure-normalization rewrites near type syntax/notation.
    For one-off constructor goals, `change` may be acceptable, but repeated
    occurrences should become a normalizer lemma.

5. `relevant_env` / `relevant_lvars` compatibility folds

- Examples:
  - `TypeDenote.v` Wand/Arrow equivalence support;
  - `TypePersistArrow.v`, `TypePersistWandForward.v`.
- Cause:
  - Some older lemmas still expect folded `relevant_env` notation while newer
    definitions expose direct `lty_env_restrict_lvars`.
- Fix pattern:
  - Prefer direct support expressions in lemma statements; use normalization
    lemmas at the boundary, not `change` inside proof bodies.

6. Application of stdpp lemmas through wrappers

- Examples:
  - `change (ext_out Fx ## world_dom ...)`;
  - `change (x ∈ dom ...)`.
- Cause:
  - Regular facts produce a wrapper-level statement, then stdpp lemmas need a
    map/set-level statement.
- Fix pattern:
  - Add folder-local regular facts that directly expose the stdpp shape from
    repo definitions.
