# Assert / Pose Proof Cleanup Plan

## Purpose

This optimization pass audits long proof scripts for repeated `assert` and
`pose proof` chains.  The goal is to make semantic steps explicit and move
repeated side-condition facts into named lemmas or local deterministic tactics.

This pass does not delete lemmas.  Short one-use wrappers can remain; the main
goal is to avoid repeating the same proof fragments inside large lemmas.

## Skill Guidance Used

- Repeated `assert`/`pose` patterns should become lower-layer lemmas.
- Inline very short one-use wrappers unless they name a real semantic boundary
  or improve compile time.
- If a helper grows hard to read, reassess and split it.
- New helpers should be semantic, normalized, and not overly shape-specific.

## Classification Rules

For every `assert` or `pose proof`, classify it as one of:

1. Semantic checkpoint.
   - Keep it if it names a real stage of the proof.
   - Example: opened argument denotation, result body, child context.

2. Repeated side condition.
   - Extract a lemma or add to a local tactic.
   - Example: freshness from a big union, domain subset, closed-on facts.

3. Raw set/map/store fact.
   - Lower to Store/Algebra/DenotationSetMapFacts.

4. Normalization residue.
   - Fix the proof boundary normalizer instead.
   - Example: raw `formula_open`, `lty_env_open_one`,
     `cty_open 0 z (cty_shift 0 τ)`.

5. One-use local plumbing.
   - Inline with `eapply`, `rewrite`, or `specialize` if it makes the proof
     shorter and does not hurt compile time.

## Concrete Optimization Work

1. Generate a local audit for each target file.

   ```sh
   rg -n "\\bassert \\(|pose proof" FILE
   ```

2. Extract only when the new lemma is more general or shorter than the local
   block.

3. Normalize new lemma statements before extraction.
   - No raw opened `ty_denote_gas` if a normalized body abbreviation exists.
   - No raw `formula_open 0 z` if `over_open_body`/`under_open_body` applies.
   - No raw typed binder environment if inserted free-variable env can be used.

4. Keep semantic phase assertions.
   - For long induction proofs, named checkpoints improve reviewability.
   - Do not mechanically eliminate every `assert`.

5. Update comments near retained long assertions.
   - Explain why they are semantic checkpoints or compile-time cuts.
