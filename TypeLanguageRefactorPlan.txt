ContextTypeLanguage cleanup / normalization plan
===============================================

Goal
----

Make ContextTypeLanguage look more like ContextLogic: syntax files should
provide small structural lemmas plus normalization tactics for fv/open/mopen,
shift, erase, and context-domain computations.  Store-only material should live
in ContextStore, and TypeLanguage-specific files should be thin where possible.

Scan summary
------------

1. `ContextTypeLanguage/theories/Qualifier.v`
   - The type qualifier syntax belongs here.
   - The block beginning with `lstore_shift_from`, `lstore_on_shift_from`, and
     the associated lstore/open-back/open-front lemmas is store-only except for
     the value specialization.  This block has been moved to ContextStore as a
     generic `V` lemma family; qualifier shift lemmas now use the lower-level
     names through the Store export.

2. `ContextTypeLanguage/theories/MultiOpen.v` and
   `ContextTypeLanguage/theories/TypeOpen.v`
   - `MultiOpen.v` is only generic finite-map opening infrastructure and is
     imported only by `TypeOpen.v`.
   - Merge it into `TypeOpen.v`; keep one syntax-only opening file.

3. `ContextTypeLanguage/theories/LtyEnv.v`
   - It is currently a thin TypeLanguage compatibility layer over generic
     `ContextStore` lvar-store operations.
   - Many lemma wrappers are one-line `apply lvar_store_*`.  Future cleanup
     should replace upper call sites with generic names when it does not hurt
     readability.  For now keep public compatibility names that are still used
     broadly.

4. `ContextTypeLanguage/theories/WF.v`
   - The initial `lvar_wf_at`, `lvars_wf_at`, `lvars_lc_at` block was not
     type-specific and has been moved to ContextBase.
   - The `basic_context_ty_lvars` / `wf_context_ty_at` block is type syntax and
     should stay in TypeLanguage.
   - Several long proofs manually normalize `fv_cty`, `context_ty_lvars_at`,
     `cty_open`, and `erase_ty`; these should use TypeLanguage syntax tactics.

5. Existing tactics
   - `ContextTypeLanguage/theories/Notation.v` has lightweight
     `mopen_norm`, `type_lvars_norm`, and `type_env_norm`, but they are placed
     in a public notation file and are too small to cover syntax proof shapes.
   - `Denotation/theories/ContextTypeDenotationTactics.v` has denotation-level
   FV normalization.  The pure pieces (context type fv/open/shift/erase and
     context dom/fv/stale) are now available from TypeLanguage first; the
     denotation FV tactic now calls the TypeLanguage normalizer before its
     denotation-specific rewrites.

Execution rounds
----------------

Round A: done
  Merge `MultiOpen.v` into `TypeOpen.v`, remove it from `_CoqProject`, and
  delete the standalone file.

Round B: done
  Add pure syntax normalizers to `Syntax.v`:
  `cty_lvars_syntax_norm`, `cty_fv_syntax_norm`, `cty_open_syntax_norm`,
  `cty_shift_syntax_norm`, `cty_erase_syntax_norm`, `ctx_syntax_norm`, plus
  `_in` variants and a combined `type_syntax_norm`.

Round C: done
  Add mopen/open-env normalizers to `TypeOpen.v`:
  `cty_open_env_syntax_norm`, `qual_open_env_syntax_norm`,
  and combined `type_open_env_syntax_norm` with `_in` variants.

Round D: done
  Point the old lightweight tactics in `Notation.v` at the new syntax tactics
  rather than duplicating small rewrites there.

Round E: done
  Compile ContextTypeLanguage, then full `make -j2`, then commit.
