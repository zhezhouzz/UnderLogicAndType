TypeEquivWand Optimization Analysis
===================================

Goal
----
Reduce repeated proof script in `Denotation/theories/TypeEquivWand.v` without
changing theorem statements or semantics.

Implemented
-----------

1. Removed pure forwarding wrappers:

- `ty_equiv_wand_result_src_mid_inserted`
  - It was definitionally the same proof as `ty_equiv_wand_result_src_mid`.
  - The only caller now calls `ty_equiv_wand_result_src_mid` directly.

- `basic_value_has_ltype_wand_inserted_result_target_arg`
- `basic_tm_has_ltype_wand_inserted_result_target_fun`
  - Both only forwarded to the corresponding Arrow helper.
  - Internal Wand proof sites now use the Arrow helpers directly.

2. Factored repeated app equivalence/totality construction:

- `wand_open_result_apps_equiv_total_product_right`
  - Covers the “open argument world, then product on the right” case.
  - Replaces repeated construction of base equivalence, product equivalence,
    and tapp equivalence/totality.

- `wand_open_result_apps_equiv_total_product`
  - Covers the direct product case.
  - Replaces the same repeated proof shape in product-target lemmas.

3. Removed dead intermediate assertions:

- `Heq_apps` in both zero-result target proofs was not used after construction.
  It has been removed.

Deferred
--------

- `basic_world_formula_wand_open_result_target` and product variant still have
  repeated context-WF/basic-world extraction. These are semantic guard assembly
  lemmas, so further factoring should be done together with the Arrow file.

- `ty_denote_gas_zero_wand_open_result_target` and product variant still share
  the final guard reconstruction pattern. This can be factored once the same
  pattern is checked in `TypeEquivArrow.v`, otherwise we risk moving complexity
  sideways.

Verification
------------

- `Denotation/theories/TypeEquivWand.vo`: passed.
- `ContextTyping/theories/SoundnessLamWand.vo`: passed.
