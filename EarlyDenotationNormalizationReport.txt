# Early Denotation Normalization Report

This report records the remaining places where opened denotation, erasure, and
relevant-environment expressions are normalized too late.  The common problem is
that helper lemmas expose raw syntax such as `formula_open 0 y (...)`,
`lty_env_open_one 0 y (typed_lty_env_bind ...)`, or
`relevant_env (lty_env_open_one 0 y ...) ...` in their premises.  Those
expressions should be normalized at the caller boundary, especially at the start
of Fundamental case proofs, before extracting reusable helpers.

The intent is not to delete information.  Pure normalization should use
equality-preserving rewrites.  Side-condition solving may use solver wrappers
such as `better_set_solver`, but solver-only facts should not become part of the
normalizer itself unless the rewrite is an equality.

## Root Cause

The current Lam/Arrow/LamD/Wand proof support still follows an older pattern:

1. Open the type denotation formula.
2. Pass the raw opened formula into a helper lemma.
3. Inside the helper, rewrite `formula_open 0 y (ty_denote_gas ...)`,
   simplify `open_tm 0 (vfvar y) (tret (vbvar 0))`, rewrite
   `cty_open_preserves_erasure`, and only then transport environments.

This makes helper statements long and brittle.  It also forces later helpers to
repeat relevant-env and erasure side conditions that should have been solved
once at the proof boundary.

The preferred pattern is:

1. Immediately after opening a branch of the denotation, assert the freshness
   packet for the opened variable.
2. Run a narrow opened-denotation normalizer.
3. Pass only the normalized `ty_denote_gas` / `ty_guard_formula` premises to
   helper lemmas.

When it also ran `lty_env_open_bind_norm`, `TypeEquivArrow.vo` exceeded the
50s single-file timeout.  Keep denotation opening and environment bind
normalization as separate steps unless a proof locally needs both.

## File-by-File Findings

### `ContextTyping/theories/Soundness.v` and `SoundnessLam.v`

These two files should be read together.  `SoundnessLam.v` is the helper file,
but the raw opened formulas are first produced by `Soundness.v`:

- `fundamental_lam_case` unfolds `ty_denote_under` / `ty_denote_gas`, enters the
  Arrow branch, and receives the opened argument premise from
  `res_models_impl_intro`.
- `fundamental_lamd_case` unfolds the `CTWand` denotation, enters the `FBWand 1`
  branch, destructs the singleton opening environment, and receives the opened
  argument premise from the FBWand argument assumption.

Those are the earliest correct normalization boundaries.  `SoundnessLam.v`
currently accepts the raw premises from `Soundness.v` and normalizes them inside
helper proofs, which is why the helper statements remain large.  The main fix is
therefore two-sided:

1. Normalize in `Soundness.v` immediately after the Arrow/FBWand branch produces
   the opened premise.
2. Change `SoundnessLam.v` helper statements so they consume the normalized
   premise rather than repeating the opening/normalization work.

So the problem is not that `SoundnessLam.v` alone is wrong; it is a caller/callee
interface mismatch.  `Soundness.v` should stop passing raw opened formulas, and
`SoundnessLam.v` should stop asking for them.

#### `lam_arrow_opened_app_static_guard_full`

Current conclusion exposes:

```coq
ty_static_guard_formula
  (lty_env_open_one 0 y
    (typed_lty_env_bind
      (atom_env_to_lty_env (erase_ctx Γ)) (erase_ty τx)))
  (cty_open 0 y τ)
  (tapp_tm (tret (vlam (erase_ty τx) e)) (vfvar y))
```

The proof first builds a static guard under:

```coq
atom_env_to_lty_env (erase_ctx (CtxComma Γ (CtxBind y τx)))
```

and then rewrites it to the opened typed-bind environment.  This rewrite should
not be exposed as the public helper shape.  Preferred repair:

- State the helper in the context-erasure form, or immediately normalize the
  context-erasure form at the call site.
- If a downstream lemma needs the opened typed-bind env, perform the rewrite in
  that local call, not in the top-level static-guard helper statement.

This avoids making every later helper mention the long
`lty_env_open_one 0 y (typed_lty_env_bind ...)` expression.

#### `lam_wand_opened_app_static_guard_full`

Same issue as the Arrow version, using `CtxStar Γ (CtxBind y τx)`.

Preferred repair:

- State the helper against
  `atom_env_to_lty_env (erase_ctx (CtxStar Γ (CtxBind y τx)))`.
- Use `atom_env_to_lty_env_erase_ctx_star_bind_open_one` only when a normalized
  opened typed-bind environment is explicitly required.

#### `lam_arrow_open_arg_to_bind_denotation`

Current premise exposes:

```coq
my ⊨ formula_open 0 y
  (ty_denote_gas ... (typed_lty_env_bind
    (relevant_env (atom_env_to_lty_env (erase_ctx Γ))
      (CTArrow τx τ) (tret (vlam ...)))
    (erase_ty τx))
    (cty_shift 0 τx) (tret (vbvar 0)))
```

The proof immediately proves freshness facts, rewrites
`formula_open_ty_denote_gas_singleton`, changes `open_tm ...` to
`tret (vfvar y)`, saturates gas, transports the relevant env, and rewrites
`cty_open 0 y (cty_shift 0 τx) = τx`.

This is exactly the normalization that should happen before calling this
helper.  Preferred new statement:

```coq
my ⊨ ty_denote_gas ...
  (lty_env_open_one 0 y
    (typed_lty_env_bind
      (relevant_env (atom_env_to_lty_env (erase_ctx Γ))
        (CTArrow τx τ) (tret (vlam ...)))
      (erase_ty τx)))
  (cty_open 0 y (cty_shift 0 τx)) (tret (vfvar y)) ->
...
```

Then a second, smaller helper can transport this to the full context-erasure
bind denotation.  The caller should use `ty_denote_open_one_norm_in` once.

#### `lam_arrow_open_arg_to_bind_ctx`

This inherits the raw `formula_open 0 y` premise from
`lam_arrow_open_arg_to_bind_denotation`.  Once the denotation helper takes a
normalized premise, this lemma should also take the normalized premise.

#### `lam_arrow_open_arg_to_comma_ctx`

Same inheritance as `lam_arrow_open_arg_to_bind_ctx`.  It should only compose:

- closure context,
- normalized opened argument denotation,
- bind-context construction,
- comma intro.

It should not know how to open `ty_denote_gas`.

#### `lam_arrow_result_mid_app_guard`

Current conclusion still contains:

```coq
ty_guard_formula
  (relevant_env
    (lty_env_open_one 0 y
      (typed_lty_env_bind
        (atom_env_to_lty_env (erase_ctx Γ)) (erase_ty τx)))
    (cty_open 0 y τ)
    (tapp_tm ...))
```

This is not fully normalized at the boundary.  The helper currently builds
`ty_static_guard_relevant_of_full` internally.  Better shape:

- Either take a full static guard and return a relevant guard internally, but do
  not expose the raw `relevant_env (lty_env_open_one ...)` expression in the
  statement.
- Or define a short canonical opened-app environment alias/equality and ensure
  all callers normalize to that alias first.

Since notation/aliases have been aggressively cleaned in this repo, the first
option is preferable: keep the statement semantic (`Hstatic_app`, `Hzero_body`)
and hide the relevant-env restriction as a local proof step.

#### `lam_arrow_result_mid_app_zero`

This consumes `lam_arrow_result_mid_app_guard`; once that helper is simplified,
this lemma should no longer need to mention the raw relevant env.  It should
only combine:

- normalized body zero denotation,
- static app guard,
- term result equivalence between `(e ^^ y)` and `((λ.e) y)`.

#### `lam_arrow_result_mid_app_denotation`

Same issue: it threads the raw opened argument premise and raw opened app env.
After repairs it should consume:

- normalized opened argument denotation,
- body denotation under normalized opened env,
- app zero guard.

#### `lam_arrow_result_mid_to_opened_goal`

This helper transports from the mid/full opened env to the relevant target env.
It should remain a semantic environment-transport helper, but callers should
not feed it unnormalized `formula_open` premises.

#### `lam_body_to_opened_arrow_result` and `lam_opened_arrow_result`

These are the current helper boundary points, but the first normalization should
move one level earlier into `Soundness.v`'s `fundamental_lam_case`.  After
`res_models_impl_intro` receives the Arrow argument premise, normalize the opened
antecedent before calling `lam_opened_arrow_result`.  `lam_opened_arrow_result`
may still do local result-side normalization, but it should not receive a raw
argument formula.

Recommended local pattern:

```coq
assert (Hτa_fresh : y ∉ fv_cty (cty_shift 0 τx)) by ...
assert (Hea_fresh : y ∉ fv_tm (tret (vbvar 0))) by ...
pose proof Harg as Harg_norm.
ty_denote_open_one_norm_in Harg_norm.
```

For the consequent, normalize after rewriting `formula_open_impl` and before
constructing app/result helpers.

#### LamD helpers

The same raw-shape pattern appears in:

- `lamd_open_arg_to_star_ctx`
- `lamd_body_to_opened_wand_result`
- `lamd_opened_wand_result`

The Wand/FBWand route should normalize its opened argument denotation in
`Soundness.v`'s `fundamental_lamd_case`, immediately after:

```coq
rewrite !formula_open_env_singleton in Harg |- *.
```

At that point the singleton opening is known and the caller has the right
freshness context.  The normalized argument should then be passed to
`lamd_opened_wand_result` and downstream bind/star helpers.

### `Denotation/theories/TypeEquivArrow.v`

This file is cleaner than `SoundnessLam.v` in the argument branch but still has
several late-normalization remnants.

#### `ty_denote_gas_tm_equiv_arrow_open_arg`

Already partially repaired: it takes normalized opened `ty_denote_gas`
premises/conclusions.  This is the style SoundnessLam should imitate.

Remaining issue: the premises still expose
`lty_env_open_one 0 y (typed_lty_env_bind (relevant_env ...) ...)`.  That is
acceptable for a low-level transport lemma, but not ideal for Soundness-level
case helpers.

#### `basic_world_formula_opened_arg_from_denotation`

Still takes:

```coq
my ⊨ formula_open 0 y
  (ty_denote_gas gas Σarg (cty_shift 0 τx) (tret (vbvar 0)))
```

This lemma should be split:

- A normalized version taking
  `my ⊨ ty_denote_gas gas (lty_env_open_one 0 y Σarg)
     (cty_open 0 y (cty_shift 0 τx)) (tret (vfvar y))`.
- A thin wrapper, if needed, that opens the formula using
  `ty_denote_open_one_norm_in`.

The normalized version should be used by future proofs.

#### Result helpers

Several Arrow result helpers still take or return `formula_open 0 y` for the
opened result.  These should be normalized at the caller after
`formula_open_impl`, rather than inside each helper.  The existing
`ty_equiv_arrow_result_src_mid` and `ty_equiv_arrow_result_tgt_goal` are good
low-level env-agree lemmas; the surrounding opened-result helpers should be
converted to consume normalized denotations.

### `Denotation/theories/TypeEquivWand.v`

This file has the same issues as Arrow, with more occurrences because both old
Wand and FBWand transitional helper shapes are still present.

#### `ty_denote_gas_tm_equiv_wand_open_arg`

Still takes raw `formula_open 0 y` premises and opens internally.  It should
match the already-repaired Arrow helper shape:

```coq
n ⊨ ty_denote_gas gas
  (lty_env_open_one 0 y
    (typed_lty_env_bind
      (relevant_env Σ (CTWand τx τr) e2)
      (erase_ty τx)))
  (cty_open 0 y (cty_shift 0 τx)) (tret (vfvar y)) ->
...
```

#### `ty_denote_gas_tm_equiv_wand_open_arg_fbwand`

Same raw `formula_open` issue.  Prefer one normalized core lemma and a small
compat wrapper only if both call sites still need it.

#### Result helper family

Many result helpers expose:

- `formula_open 0 y (...)`,
- `relevant_env (lty_env_open_one 0 y (typed_lty_env_bind Σ ...)) ...`,
- explicit `cty_open_preserves_erasure`.

The same fix as Arrow applies:

- normalize opened denotations once at the FBWand branch boundary,
- use `wand_body_relevant_env_agree_open_one_core` at the boundary,
- keep result helpers as env-agree or term-equivalence transport only.

### `Denotation/theories/TypeDenote.v`

The new tactics are in the right layer, but should remain narrowly scoped:

- `ty_denote_open_one_norm(_in)` should only normalize opened denotation
  syntax and obvious LN term openings.
- `lty_env_open_bind_norm(_in)` should stay separate because automatic use
  caused `TypeEquivArrow.vo` timeout.
- A future `ty_denote_opened_impl_norm(_in)` can be added as a small composed
  tactic:
  rewrite `formula_open_impl`, then run `ty_denote_open_one_norm(_in)` on the
  antecedent/consequent.  It must not run broad store/env normalizers by default.

### `ContextBasicDenotation/theories/RelevantEnv.v`

This file has the right core lemmas, but they are being applied too late in
Soundness-level proofs.

Most useful existing facts:

- `arrow_arg_relevant_env_agree_open_one_core`
- `arrow_body_relevant_env_agree_open_one_core`
- `wand_arg_relevant_env_agree_open_one_core`
- `wand_body_relevant_env_agree_open_one_core`
- `context_ty_wf_formula_relevant_env`
- `context_ty_wf_formula_relevant_env_change_term`

Recommended addition, if repeated proofs remain noisy:

```coq
Ltac opened_relevant_env_agree :=
  (* choose arrow/wand arg/body relevant-env agree lemma by goal shape *)
```

This should be a solver, not a pure normalizer, because it uses subset/agreement
facts rather than definitional equalities.

### `ContextTypeLanguage/theories/Syntax.v`

The erasure rewrite lemmas are already available:

- `cty_open_preserves_erasure`
- `erase_ctx_comma_bind_dom`
- `erase_ctx_star_bind_dom`
- `erase_ctx_comma_bind_fresh`
- `erase_ctx_star_bind_fresh`

The problem is not absence of lemmas; it is that high-level proofs still call
them manually inside helper proofs.  `cty_erase_syntax_norm(_in)` and
`ctx_syntax_norm(_in)` should be called before helper extraction.

## Recommended Refactor Order

1. Add a small tactic:

```coq
Ltac ty_denote_opened_impl_norm :=
  rewrite ?formula_open_impl;
  ty_denote_open_one_norm;
  type_open_env_syntax_norm.

Ltac ty_denote_opened_impl_norm_in H :=
  rewrite ?formula_open_impl in H;
  ty_denote_open_one_norm_in H;
  type_open_env_syntax_norm_in H.
```

Keep it narrow.  Do not include `lty_env_open_bind_norm`.

2. In `Soundness.v`, normalize the Arrow argument premise in
   `fundamental_lam_case` before calling `lam_opened_arrow_result`.

3. Change `lam_opened_arrow_result` and then
   `lam_arrow_open_arg_to_bind_denotation` to take a normalized opened
   denotation premise.  `lam_arrow_open_arg_to_bind_denotation` should no longer
   call `formula_open_ty_denote_gas_singleton`.

4. Propagate the normalized premise through:

- `lam_arrow_open_arg_to_bind_ctx`
- `lam_arrow_open_arg_to_comma_ctx`
- `lam_body_to_opened_arrow_result`
- `lam_opened_arrow_result`

5. Change `lam_arrow_opened_app_static_guard_full` and
   `lam_wand_opened_app_static_guard_full` to avoid exposing opened typed-bind
   envs in their statements.  Prefer context-erasure envs at the statement
   boundary.

6. Simplify `lam_arrow_result_mid_app_guard` so its statement does not expose
   `relevant_env (lty_env_open_one 0 y (typed_lty_env_bind ...))`.  Either:

- accept full static guard plus body zero denotation and build the relevant
  guard internally, or
- accept an already normalized `ty_guard_formula` premise produced by a caller
  normalizer.

7. Repeat the same transformation for Wand/LamD, starting in `Soundness.v`'s
   `fundamental_lamd_case`:

- normalize FBWand opened argument at the FBWand eliminator boundary,
- pass normalized denotations to bind/star helpers,
- keep `formula_open 0 y` out of helper statements unless the helper is
  explicitly an opening lemma.

8. Convert low-level TypeEquivWand open-arg helpers to the same normalized
   shape already used by `ty_denote_gas_tm_equiv_arrow_open_arg`.

## Compile-Time Guardrails

- Use 50s single-file checks after each batch.
- Do not add broad `repeat rewrite` normalizers to global tactics unless they
  are equality-only and have already been tested on `TypeEquivArrow.vo` and
  `SoundnessLam.vo`.
- Prefer explicit freshness packets:

```coq
assert (Hτ_fresh : y ∉ fv_cty (...)) by ...
assert (He_fresh : y ∉ fv_tm (...)) by ...
assert (HΣ_fresh : y ∉ lvars_fv (dom ...)) by ...
```

Then run `ty_denote_open_one_norm_in H`.

- Keep `lty_env_open_bind_norm` explicit.  It is valuable, but too expensive to
  run automatically inside every denotation-opening normalizer.

## Expected Result

After this pass, Soundness-level Lam/LamD helpers should not expose:

- raw `formula_open 0 y (ty_denote_gas ...)` premises,
- raw `lty_env_open_one 0 y (typed_lty_env_bind ...)` static-guard statements,
- raw `relevant_env (lty_env_open_one 0 y (typed_lty_env_bind ...))` in major
  app/result guard helper statements,
- manual `cty_open_preserves_erasure` and `erase_ctx_*_bind_*` blocks inside
  extracted helper statements.

Those expressions may still appear in low-level normalization or env-agree
lemmas, where they are the actual subject of the lemma.  They should not leak
into high-level Fundamental proof helper interfaces.

## 2026-06-05 Status

Implemented the first normalization-boundary repair.

Completed:

- `fundamental_lam_case` now converts the Arrow argument premise from raw
  `formula_open 0 y (ty_denote_gas ...)` into normalized opened
  `ty_denote_gas` before calling `lam_opened_arrow_result`.
- `fundamental_lamd_case` does the analogous conversion immediately after the
  singleton `formula_open_env` rewrite in the `FBWand 1` branch.
- `lam_opened_arrow_result`, `lam_arrow_open_arg_to_comma_ctx`,
  `lam_arrow_open_arg_to_bind_ctx`, and
  `lam_arrow_open_arg_to_bind_denotation` now consume normalized argument
  denotation premises.
- `lamd_opened_wand_result`, `lamd_open_arg_to_star_ctx`, and
  `lamd_wand_open_arg_to_bind_denotation` now consume normalized argument
  denotation premises.
- Removed unused raw argument premises from Arrow and Wand result-side helpers.

Current remaining `formula_open 0 y` occurrences in `SoundnessLam.v` are
result/consequent adapters: they produce the formula shape required by the
opened Arrow/FBWand connective goal.  They are no longer argument-side premises

## 2026-06-05 Final Follow-Up

Completed the remaining normalization leak that showed up after review:

- `Soundness.v` no longer carries the long raw argument-normalization proof
  blocks for Lam/LamD.  It calls:
  - `lam_arrow_open_arg_normalize`
  - `lamd_wand_open_arg_normalize`
- `SoundnessLam.v` now owns those normalization proofs, so the Fundamental
  dispatch file starts from shorter normalized argument denotations.
- The old mixed-purpose `lamd_body_to_opened_wand_result` helper was removed.
  Its responsibilities are split into:
  - `lamd_body_to_wand_result_mid`, which returns the normalized inserted-env
    app denotation;
  - `lamd_wand_result_mid_to_opened_goal`, which is the only LamD-specific
    adapter back to the raw `formula_open` consequent required by `FBWand`.
- `lamd_body_to_wand_result_mid` no longer exposes
  `formula_open 0 y (ty_denote_gas ...)` or
  `lty_env_open_one 0 y (typed_lty_env_bind ...)` in its statement.
- `lam_arrow_open_arg_to_bind_denotation`,
  `lam_arrow_open_arg_to_bind_ctx`, `lam_arrow_open_arg_to_comma_ctx`,
  `lam_opened_arrow_result`, `lamd_wand_open_arg_to_bind_denotation`,
  `lamd_open_arg_to_star_ctx`, and `lamd_opened_wand_result` all consume
  normalized inserted-env argument denotation premises.
