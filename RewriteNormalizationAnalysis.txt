# Rewrite Normalization Analysis

This report counts and classifies explicit set/erase/FV/relevant-env rewrites
in `ContextBasicDenotation`, `Denotation`, and `ContextTyping`.  The purpose is
to decide which rewrites should be absorbed into normalization tactics and
which should remain explicit semantic proof steps.

## Pure Equality Rewrites To Add To Normalization

These are safe for early normalization because they preserve equality and do
not consume semantic hypotheses.

### Type/term FV and lvars

Add to `relevant_lvars_norm` and `_in`:

- `fv_tapp_tm`
- `tm_shift_fv`
- `cty_shift_fv`
- `lvars_at_depth_union`
- `lvars_at_depth_empty`
- `lvars_fv_empty`
- `lvars_fv_singleton_bound`
- `lvars_fv_singleton_free`
- `lvars_fv_of_atoms`
- `typed_lty_env_bind_lvars_fv_dom`
- `lvar_store_bind_lvars_fv_dom`
- `dom_insert_L`
- `dom_empty_L`
- `dom_union_L`

Some of these already live in other normalizers.  The goal is to make
`relevant_lvars_norm` call those normalizers or cover their common outputs, so
upper proofs do not need to remember which lower tactic owns which rewrite.

### Type erasure and open/shift

Already partly covered by `type_syntax_norm`; ensure relevant-env proofs call
it early:

- `cty_open_preserves_erasure`
- `cty_shift_preserves_erasure`
- `open_cty_env_preserves_erasure`
- `cty_open_preserves_depth`
- `cty_shift_preserves_depth`

These should stay in pure syntax normalization, not in a side-condition solver.

### Context erasure

Already partly covered by `ctx_syntax_norm`; ensure `ctx_erasure_under_norm`
also rewrites the context-erasure shape lemmas:

- `ctx_erasure_under_minimal`
- `ctx_erasure_under_bind`
- `ctx_erasure_under_comma`
- `ctx_erasure_under_star`
- `erase_ctx_comma_bind_dom`
- `erase_ctx_star_bind_dom`
- `erase_ctx_comma_bind_fresh`
- `erase_ctx_star_bind_fresh`

`ctx_erasure_under_bind_closed` has a nontrivial closedness premise and should
not be run blindly by pure normalization.  It can be used in a solver wrapper
when `basic_context_ty ∅ τ` is available.

### Opened typed-bind environments

Add explicit equality lemmas instead of repeating this chain:

```coq
rewrite typed_lty_env_bind_open_current.
rewrite <- atom_store_to_lvar_store_insert.
rewrite erase_ctx_comma_bind_fresh.
```

Suggested lemmas:

```coq
atom_env_to_lty_env_erase_ctx_comma_bind_open_one
atom_env_to_lty_env_erase_ctx_star_bind_open_one
```

Then add a solver-oriented tactic:

```coq
lty_env_open_bind_norm
lty_env_open_bind_norm_in
```

This tactic may use side-condition solving (`eauto`, `better_set_solver`,
`atom_store_to_lvar_store_closed`) because `typed_lty_env_bind_open_current`
requires freshness/closedness premises.

## Non-Equivalence Or Side-Condition Rewrites

These should not be placed in pure normalization.  Put them in solver wrappers
or use them explicitly.

- `typed_lty_env_bind_open_current`: needs freshness and closedness.
- `relevant_env_open_env`: needs open-env freshness and injectivity.
- `relevant_env_open_one`: needs opened variable freshness.
- `tm_lvars_tapp_tm_fvar_without_arg`: depends on LC/body/fresh facts.
- `tm_lvars_tapp_tm_fvar_without_arg_shift_lc`: depends on LC facts.
- `ctx_erasure_under_bind_closed`: depends on closed type.
- `basic_context_ty_*` inversions: semantic/wf facts, not syntax facts.

## File-by-File Recommendations

### ContextBasicDenotation/theories/RelevantEnv.v

This is the right home for:

- richer relevant-lvars normalization;
- opened typed-bind env normalization;
- `relevant_env_set` for side-condition goals.

Do not overload `relevant_lvars_norm` with non-equivalence transformations.
Keep those in `relevant_env_set` or a new `relevant_open_env_solver`.

### Denotation/theories/TypeDenote.v

Replace local unfold/set blocks in open-env transport and env-agree proofs with
`relevant_lvars_norm` / `relevant_env_set`.

Potentially replace local `normalize_denot_formula_lvars` with a call to a
shared lvars normalizer plus denotation-specific `lvars_at_depth` rewrites.

### Denotation/theories/TypeEquivCore.v

Repeated term-app FV/lvars patterns should be normalized by a term syntax
normalizer.  Do not alter semantic equivalence theorem statements.

### Denotation/theories/TypeEquivArrow.v
### Denotation/theories/TypeEquivWand.v

Use `relevant_lvars_norm` for:

- `cty_shift_fv`;
- `fv_tapp_tm`;
- `tm_shift_fv`;
- `typed_lty_env_bind_lvars_fv_dom`;
- `relevant_env_dom_subset_direct` side conditions.

Opened-result proofs should still keep their semantic packages explicit.

### Denotation/theories/Context.v

Keep `ctx_erasure_under_*` semantic lemmas.  Strengthen
`ctx_erasure_under_norm` so callers no longer unfold `ctx_erasure_under` unless
proving those lemmas themselves.

### ContextTyping/theories/SoundnessLam.v

Highest payoff:

1. Use opened typed-bind environment equality lemmas in the two static guard
   proofs.
2. Use a full-env static guard plus a relevant-guard transport lemma to stop
   propagating giant `relevant_env` premises.
3. Use `relevant_lvars_norm` in fresh side-condition packets.

### ContextTyping/theories/Soundness.v

Mostly already uses new normalizers.  Remaining explicit `relevant_lvars_fv`
or `ctx_erasure_under` rewrites should be converted after the normalizers are
extended.

## Implementation Plan

1. Add opened typed-bind environment equality lemmas and tactic support.
2. Extend `relevant_lvars_norm` with the pure equality rewrites listed above.
3. Extend `ctx_erasure_under_norm` with the context erasure shape lemmas.
4. Refactor the two opened app static guard proofs to use the new environment
   equality lemmas.
5. Add full-env static guard and relevant static guard transport, then thin
   the `lam_*_app_*` helper premises if compilation remains fast.
