# Fiberwise / msubst route cleanup analysis

This report audits the current branch after the direct Fundamental route was
restored.  The goal is to identify proof code that belongs to the abandoned
fiberwise/msubst route, placeholder case bridges that should be removed or
restated, and useful new infrastructure that should be kept.

## 1. Current proof state

Top-level `Fundamental` in `ContextTyping/theories/Soundness.v` is direct and
`Qed`.  The already-restored direct cases are:

- `Var`
- `Const`
- `Sub`
- `CtxSub`
- ordinary `Let`
- `AppOp`

The ordinary `Let` case now calls `tlet_intro_denotation` directly and no
longer depends on the old result-extension context admit.  The helper
`ctx_comma_bind_of_result_ext` is proved.

The remaining `Soundness.v` admits are not part of this restored core route:

- `ty_denote_under_fiber_elim_projection_instantiated_from_wf`
  Old fiberwise compatibility obligation.
- `ctx_denote_under_local_relevant_basic_world`
  Looks obsolete after the current direct static-guard proof.  It should be
  checked for uses and then deleted if unused.
- corollaries: `safety`, `coverage`, `refinement`, `incorrectness`,
  `exact_result`.

Higher-order and branching cases still dispatch through placeholder bridge
lemmas in `Denotation/theories/DenotationCases.v`:

- `letd_soundness_bridge`
- `lam_soundness_bridge`, `lamd_soundness_bridge`
- `app_soundness_bridge`, `appd_soundness_bridge`
- `fix_soundness_bridge`, `fixd_soundness_bridge`
- `match_both_soundness_bridge`, `match_true_soundness_bridge`,
  `match_false_soundness_bridge`

The direct semantic obligations behind those bridges are also admitted:

- `letd_intro_denotation`
- `lam_intro_denotation`, `lamd_intro_denotation`
- `app_elim_denotation`, `appd_elim_denotation`
- `fix_intro_denotation`, `fixd_intro_denotation`
- `match_both_intro_denotation`, `match_true_intro_denotation`,
  `match_false_intro_denotation`
- `appop_intro_denotation_msubst` appears to be an extra/old msubst-flavored
  obligation, not needed by current `fundamental_appop_case`.

## 4. Delete/restatement candidates: placeholder case bridges

The current `Cases.v` bridges were introduced to keep `Fundamental` compiling
while the semantic cases were not proved.  Since you plan to prove the cases by
a better method, these bridge lemmas should not become the permanent API.

Candidates to remove or replace with new direct statements:

- `letd_soundness_bridge`
- `lam_soundness_bridge`, `lamd_soundness_bridge`
- `app_soundness_bridge`, `appd_soundness_bridge`
- `fix_soundness_bridge`, `fixd_soundness_bridge`
- `match_both_soundness_bridge`, `match_true_soundness_bridge`,
  `match_false_soundness_bridge`

The corresponding direct semantic obligations should also be reviewed before
keeping:

- Some names are useful as future theorem names (`lam_intro_denotation`,
  `app_elim_denotation`, etc.).
- But current statements may be too proof-specific and may encode the older
  bridge shape.  If the new proof method is different, delete the placeholders
  and reintroduce only the statements that match the new method.

`appop_intro_denotation_msubst` is a strong delete candidate: current
`fundamental_appop_case` is already direct and does not need this msubst-flavored
obligation.

## 5. Delete candidates: giant type-denotation msubst route

The public theorem that should be retired is:

- `ty_denote_gas_msubst_store_from_typing_wf_iff`
- plus wrappers:
  `ty_denote_gas_msubst_store_from_typing_wf`,
  `ty_denote_gas_msubst_store_to_typing_wf`

These live in `Denotation/theories/TypeDenoteMsubstScoped.v` and
depend on the artificial erased typing package:

- `context_typing_wf_erased`
- `wf_erased_ctx_under`
- `context_ty_wf_for_erased`

After current direct soundness restoration, this route is no longer the desired
boundary.  The msubst theorem was built for the fiberwise/base-projection
bridge and is the source of most of the large proof mass.

Files likely removable after dependencies are severed:

- `Denotation/theories/TypeDenoteMsubstScoped.v`
- `Denotation/theories/TypeDenoteMsubstCompat.v`
- most or all of `Denotation/theories/TypeDenoteMsubstGuard.v`
- most or all of `Denotation/theories/TypeDenoteMsubstKeepDomain.v`
- most of `ContextBasicDenotation/theories/MsubstTransport.v`

Do not delete the formula-level `formula_msubst_store` operation itself.  It is
now part of `FFibVars` semantics and remains meaningful at the Logic layer.

## 6. Items to keep

### 6.1 Term-result equivalence transport

Keep the `TypeEquiv*` family, especially:

- `tm_equiv_on`
- `typed_total_equiv_on`
- `ty_denote_gas_tm_result_equiv`
- `ty_denote_gas_result_alias_to_var`
- `ty_denote_gas_result_extension_to_var`

This is the useful theorem you identified: equivalent term results transport
through context-type denotation.  It is now used by
`ctx_comma_bind_of_result_ext` in `Soundness.v` through
`ty_denote_gas_result_extension_to_var`.

### 6.2 Formula-level msubst and FFibVars infrastructure

Keep small Logic-level syntax/semantics facts:

- `formula_msubst_store`
- `formula_msubst_store_restrict_fv`
- `formula_msubst_store_agree_fv`
- `res_models_formula_msubst_store_agree_fv`
- `formula_msubst_store_fv_subset`
- `formula_msubst_store_fv_restore`
- `formula_open_msubst_store_fresh`
- FFibVars semantic facts needed by `FormulaSemantics`.

These are not the bad route.  They are basic syntax/semantics support for the
current `FFibVars` definition.

### 6.3 General Algebra/Store/Logic helpers

The commits around `0bb6e85` added many bottom-layer lemmas.  Do not bulk
delete these.  Keep helpers that are general and not tied to
`ty_denote_gas_msubst_store_*`, for example:

- product/sum/pullback/resource projection facts;
- formula FV/open/scope normalization facts;
- store restrict/agreement facts;
- better solver improvements.

These are useful independent of the abandoned msubst theorem.

### 6.4 One open-denotation lemma used by Saturate

`TypeEquivArrow.v` and `...Wand.v` currently import
`TypeDenoteMsubstCore` only for:

- `formula_open_ty_denote_gas_singleton`

Move this lemma out of `MsubstCore` into a neutral file such as
`TypeDenoteOpen.v` or `TypeEquivCore.v` before
deleting the msubst files.

## 7. Dependency notes

Current imports:

- `Denotation/theories/Context.v` imports `TypeDenoteMsubst` and
  `TypeEquiv`.
- `Soundness.v` imports `TypeDenoteMsubst` only through the current
  direct import list and through `Context.v`.
- `Saturate*.v` imports `TypeDenoteMsubstCore` only for the
  singleton open lemma noted above.

Therefore a safe cleanup should:

1. Move `formula_open_ty_denote_gas_singleton` to a non-msubst file.
2. Remove `TypeDenoteMsubst` from `Context.v` and `Soundness.v`
   imports.
3. Delete or comment out all old fiberwise wrappers in `Context.v`.
4. Delete the msubst file family from `_CoqProject`.
5. Compile and only then prune BasicDenotation msubst helpers that become
   unused.

## 8. Suggested cleanup order

### Batch A: remove old fiberwise wrappers

Target files:

- `Denotation/theories/Context.v`
- `ContextTyping/theories/Soundness.v`

Actions:

- Delete `ty_denote_under_fiber` and all `_fiber` helpers.
- Delete old projection/body/fiber context helpers listed in section 3.
- Delete `ty_denote_under_fiber_elim_projection_instantiated_from_wf`.
- Compile `Denotation/theories/Context.vo` and `ContextTyping/theories/Soundness.vo`.

Risk:

- Low to medium.  Current direct Fundamental should not need these.  If a
  remaining placeholder bridge uses them, delete/restated that bridge rather
  than preserving the fiber helper.

### Batch B: neutralize the msubst import dependency

Actions:

- Move `formula_open_ty_denote_gas_singleton` out of
  `TypeDenoteMsubstCore.v`.
- Update `SaturateArrow/Wand` imports.
- Remove `TypeDenoteMsubst` imports from `Context.v` and
  `Soundness.v`.

Risk:

- Low.  The dependency scan shows `Saturate*` only needs this one lemma from
  `MsubstCore`.

### Batch C: delete msubst Denotation file family

Actions:

- Remove from `_CoqProject`:
  - `TypeDenoteMsubstCore.v`
  - `TypeDenoteMsubstGuard.v`
  - `TypeDenoteMsubstKeepDomain.v`
  - `TypeDenoteMsubstCompat.v`
  - `TypeDenoteMsubstScoped.v`
  - `TypeDenoteMsubst.v`
- Delete the files.
- Compile.

Risk:

- Medium.  Some old compatibility lemmas in `Context.v` and `Cases.v` may still
  reference msubst.  They should be removed, not repaired.

### Batch D: prune `MsubstTransport.v`

Actions:

- Check whether `ContextBasicDenotation/theories/MsubstTransport.v` is still
  imported only via `Denotation/theories/Notation.v`.
- Delete the erased typing wrapper definitions:
  - `wf_erased_ctx_under`
  - `context_ty_wf_for_erased`
  - `context_typing_wf_erased`
- Delete atom/guard msubst lemmas that only served the retired type-denotation
  transport.
- Keep or move any formula/term msubst helper still used by Logic or
  `Saturate`.

Risk:

- Medium.  This file has mixed content: some lemmas are purely old-route, while
  some low-level term/formula msubst facts may be useful.

### Batch E: remove placeholder case bridges

Actions:

- In `DenotationCases.v`, delete the current placeholder bridge layer
  or replace it with the new method's statements.
- Keep only proved direct case lemmas (`const_direct_denotation_gas_in_ctx`,
  `appop_intro_denotation`, etc.) and future theorem names that match the new
  proof method.

Risk:

- Medium.  `Soundness.v` currently calls these bridges for unproved cases.  If
  deleted immediately, `Fundamental` will need those branches admitted directly
  or reconnected to the new case statements.

## 9. Bottom-line recommendation

The cleanup is feasible and should substantially shrink the repo.  The best
first move is not deleting the largest file immediately; it is moving the one
useful open lemma out of `MsubstCore` and removing old fiberwise wrappers from
`Context.v`.  After that, the giant msubst file family should become isolated
and deletable.

Keep the term-equivalence/saturate route.  It is already useful, has a cleaner
semantic interpretation, and directly supports result-extension reasoning for
ordinary `Let`.
