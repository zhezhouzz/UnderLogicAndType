# Premise Normalization Analysis

This report tracks lemmas whose statements expose large syntactic expressions
that should normally be normalized before they cross a proof boundary.  The
goal is not to delete semantic assumptions; the goal is to avoid statements
whose premises contain raw `relevant_env`, opened lty-envs, or context-erasure
expressions that are only artifacts of syntax normalization.

## Summary

There are two distinct problems.

1. Some lemmas expose a fully expanded relevant environment in their statement,
   for example:

   ```coq
   relevant_env
     (lty_env_open_one 0 y
       (typed_lty_env_bind
         (atom_env_to_lty_env (erase_ctx Γ)) (erase_ty τx)))
     (cty_open 0 y τ)
     (tapp_tm (tret (vlam (erase_ty τx) e)) (vfvar y))
   ```

   These should usually take a full opened environment premise, or a semantic
   guard premise, and apply relevant-env restriction internally.

2. Some lemmas expose raw `ctx_erasure_under`, `erase_ctx`, or opened
   environment equalities in side conditions.  These are mostly fine as local
   proof obligations, but should be normalized immediately at the call site.

## ContextTyping/theories/SoundnessLam.v

This is the main hotspot.  It contains the direct Lam/LamD proofs and many
helper lemmas that were extracted while proving them.  The current proof is
valid, but several helper statements leak normalization details.

### `context_typing_wf_denot_static_guard`

Current shape returns:

```coq
m ⊨ ty_static_guard_formula
  (relevant_env (atom_env_to_lty_env (erase_ctx Γ)) τ e) τ e
```

Problem: callers that need an opened application guard inherit a large
`relevant_env` expression.  The context typing premise naturally gives a guard
for the full erased environment first; relevant restriction is a support
optimization, not part of the source-level statement.

Recommended split:

```coq
Lemma context_typing_wf_denot_static_guard_full ... :
  context_typing_wf Σ Γ e τ ->
  m ⊨ ctx_denote_under Σ Γ ->
  m ⊨ ty_static_guard_formula
    (atom_env_to_lty_env (erase_ctx Γ)) τ e.

Lemma ty_static_guard_relevant_of_full ... :
  m ⊨ ty_static_guard_formula Σ τ e ->
  m ⊨ ty_static_guard_formula (relevant_env Σ τ e) τ e.
```

Then keep the current relevant lemma as a wrapper, if still useful:

```coq
Lemma context_typing_wf_denot_static_guard ... :=
  context_typing_wf_denot_static_guard_full + ty_static_guard_relevant_of_full.
```

### `lam_arrow_opened_app_static_guard`

Current conclusion is a static guard under the large relevant opened app env.

Recommended shape:

```coq
my ⊨ ty_static_guard_formula
  (lty_env_open_one 0 y
    (typed_lty_env_bind
      (atom_env_to_lty_env (erase_ctx Γ)) (erase_ty τx)))
  (cty_open 0 y τ)
  (tapp_tm (tret (vlam (erase_ty τx) e)) (vfvar y)).
```

The relevant restriction should be applied only when building
`ty_guard_formula`, via `ty_static_guard_relevant_of_full`.

Needed supporting normalization:

- equality between
  `atom_env_to_lty_env (erase_ctx (CtxComma Γ (CtxBind y τx)))` and the opened
  typed bind environment;
- `cty_open_preserves_erasure`;
- `erase_ctx_comma_bind_fresh`;
- a tactic for these opened-bind erasure rewrites.

### `lam_wand_opened_app_static_guard`

Same issue as the Arrow version, but with `CtxStar Γ (CtxBind y τx)`.

Recommended shape mirrors the Arrow helper, with the full opened environment.
The relevant restriction should be hidden behind the same
`ty_static_guard_relevant_of_full` lemma.

### `lam_arrow_result_mid_app_guard`
### `lam_arrow_result_mid_app_zero`
### `lam_arrow_result_mid_app_denotation`
### `lam_arrow_result_mid_to_opened_goal`
### `lam_body_to_opened_arrow_result`

These five helpers currently propagate the large relevant opened app static
guard premise.  After the static-guard split above, their premises should
either:

- accept the full opened static guard and apply relevant restriction internally,
  or
- accept `ty_guard_formula` directly if the caller has already built it.

Best first refactor:

1. Change `lam_arrow_result_mid_app_guard` to take the full static guard.
2. Inside it, derive the relevant static guard by
   `ty_static_guard_relevant_of_full`.
3. Keep downstream helper statements unchanged initially, then do a second pass
   to propagate the smaller full-env premise.

### `lamd_wand_open_arg_to_bind_denotation`
### `lamd_open_arg_to_star_ctx`
### `lamd_body_to_opened_wand_result`
### `lamd_opened_wand_result`

These are the Wand/LamD analogues.  The argument premise is a genuine opened
`FBWand` antecedent, so it should keep the opened denotation premise.  However,
the proof body repeatedly normalizes:

- `typed_lty_env_bind_open_current`;
- `atom_store_to_lvar_store_insert`;
- `relevant_lvars_fv`;
- `cty_shift_fv`;
- context erasure under `CtxStar`.

The statement itself is mostly semantic enough; the optimization here is to
add opened-bind environment normalization and relevant-lvars side-condition
normalization, rather than to change all LamD helper statements.

## ContextTyping/theories/Soundness.v

This file now contains the direct Fundamental dispatch and lower-order cases.
Most of the visible normalization issues are side conditions:

- `dom (ctx_erasure_under Σ Γ)`;
- `fv_tm e ∪ fv_cty τ`;
- `relevant_lvars` subsets for `ty_denote_gas_env_agree_on`;
- fresh binder sets for Let/LetD.

Recommended action: keep theorem statements unchanged.  Use the new
normalizers at local assertion points only.  Do not push these expressions into
new global helper premises.

## Denotation/theories/TypeDenote.v

`TypeDenote.v` has two kinds of large premises.

### Open-env transport lemmas

`formula_open_env_denot_guard`, `formula_open_env_ty_denote_gas_zero`, and
`formula_open_env_ty_denote_gas` all use:

```coq
open_env_fresh_for_lvars η (dom Σ ∪ relevant_lvars τ e)
```

This is a semantic interface, not merely normalization noise.  Keep the premise.
However, the proof repeatedly unfolds `relevant_lvars` and rewrites lvars/FV
facts.  That should be delegated to `relevant_lvars_norm` and a shared
denotation-lvars normalizer.

### Env-agree lemmas

`ty_denote_gas_env_agree_on` and `res_models_ty_denote_gas_env_agree_on` expose:

```coq
relevant_lvars τ e ⊆ X
lty_env_restrict_lvars Σ1 X = lty_env_restrict_lvars Σ2 X
```

These are good semantic premises.  Do not simplify the statement.  Instead,
make branch side conditions use a relevant-lvars solver.

## Denotation/theories/TypeEquivCore.v

Most large premises are semantic packages for term-result equivalence.  Keep
them.  The cleanup target is proof-body normalization around:

- `fv_tm (tapp_tm ...)`;
- `tm_shift_fv`;
- `tm_lvars_tapp_tm_fvar_without_arg_shift_lc`;
- lty-env opened bind lookup lemmas.

Do not change the main theorem interfaces here unless a caller repeatedly
constructs the same package.

## Denotation/theories/TypeEquivArrow.v

The opened-result helper family has several large support premises, but most
are real semantic packages:

- opened argument world;
- opened result world;
- source/target zero guards;
- typed-total equivalence package.

The normalization target is the repeated opened-bind support:

- `typed_lty_env_bind_lvars_fv_dom`;
- `relevant_env_dom_subset_direct`;
- `cty_shift_fv`;
- `fv_tapp_tm`;
- `tm_shift_fv`;
- `cty_open_preserves_erasure`.

No statement change is recommended in the first pass.

## Denotation/theories/TypeEquivWand.v

Same as Arrow, with extra product-world support.  Most premises are semantic.
Do not thin them until the normalizers are in place and compile-time impact is
known.

## Denotation/theories/Context.v

`ctx_erasure_under_*` lemmas are the correct semantic boundary for context
erasure.  The statements should remain.  The proof bodies still contain manual
lookup chains and set proof blocks.  This file should export normalization
lemmas/tactics rather than have callers unfold `ctx_erasure_under`.

## ContextBasicDenotation/theories/RelevantEnv.v

This file is the right home for relevant-env normalization and opened-bind
environment support.  Current `relevant_lvars_norm` is useful but incomplete:
it does not yet cover all opened app, typed-bind, fv, and erasure rewrites that
show up in Lam/LamD and TypeEquiv proofs.

Recommended additions:

- opened comma/star bind environment equality lemmas;
- an opened typed-bind env normalizer;
- more FV/lvars rewrites in `relevant_lvars_norm`;
- keep non-equivalence side-condition solving in `relevant_env_set` or a new
  solver wrapper, not in pure normalization.

## Prioritized Refactor Order

1. Add the opened bind environment equality lemmas and normalizer support.
2. Extend relevant-lvars/FV normalization with missing pure equalities.
3. Refactor `lam_arrow_opened_app_static_guard` and
   `lam_wand_opened_app_static_guard` to return full-env static guards.
4. Add `ty_static_guard_relevant_of_full` and use it inside the app-guard
   helpers.
5. Only then propagate smaller static-guard premises through the Arrow/Wand
   result helper chain.

  (`typed_total_equiv_on`, opened argument/result denotations, and zero
  guards), not just normalization artifacts.
