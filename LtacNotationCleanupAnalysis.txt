Ltac / notation / comment cleanup analysis
=========================================

Scope
-----

This report covers the current compiled repo after the direct Soundness and
TLet/result-equivalence cleanup.  The goal is not to ban local tactics or
notations.  The goal is to identify:

  - stale comments that describe old file splits or old proof routes;
  - temporary Ltac that should be moved down, merged into existing solvers, or
    made local/private;
  - notation wrappers that are unused, misleading, duplicated, or worth
    upgrading because they occur frequently.

Main conclusion
---------------

The repo now has a reasonably clear tactic hierarchy:

  ContextBase/Tactics.v
    pure set/map normalization and solvers

  ContextBase/BaseTactics.v
    swap/open-env/base syntax normalization

  ContextStore/Store.v
    store restriction, lookup, insert, and map-key normalization

  ContextLogic/FormulaSyntax.v and FormulaSemantics.v
    formula syntax and scoped-model normalizers

  ContextTypeLanguage/Syntax.v and TypeOpen.v
    context-type syntax, lvars, open, shift, erase normalization

However, Denotation and BasicDenotation still define several local tactics that
repeat these lower-level normalizers.  Some are useful local orchestration, but
some should be merged into a denotation-specific tactic file or moved down.

High-priority cleanup
=====================

1. Move Denotation lvar/FV normalizers into a dedicated support file
--------------------------------------------------------------------

Current location:

  Denotation/theories/TypeDenote.v

Current Ltac:

  normalize_denotation_formula_fv
  normalize_denotation_formula_fv_in
  denot_lvars_norm
  denot_lvars_norm_in
  denot_lvars_set
  denot_ty_fv_norm
  denot_ty_fv_norm_in
  denot_ty_fv_set
  rewrite_tm_support
  unfold_formula_lvars_atoms
  normalize_denot_formula_lvars

Analysis:

  These tactics are not definition-specific.  They are syntax/FV support for all
  denotation files.  They rewrite the same families of lemmas as
  ContextTypeLanguage/Syntax.v and ContextLogic/FormulaSyntax.v:

    formula_fv_*
    formula_lvars_*
    relevant_lvars
    context_ty_lvars_fv
    tm_lvars_fv
    lvars_fv_union
    lvars_fv_of_atoms

  Keeping them in TypeDenote.v makes that file a mixed
  definition+tactic hub and forces unrelated denotation files to import core
  definitions just to get normalizers.

Recommended change:

  Create or reuse:

    Denotation/theories/TypeDenoteTactics.v

  Move the tactics above there, then have Definition/Open/Saturate files import
  it.  Keep the actual ty_denote_gas definition and semantic lemmas in
  TypeDenote.v.

  If the file already existed in an older route, recreate it as a small real
  tactic file, not as a wrapper export file.

3. BasicTypingFormula local tactics should be split into syntax-level and
   proof-local parts
---------------------------------------------------------------------------

Current location:

  ContextBasicDenotation/theories/BasicTypingFormula.v

Current Ltac:

  rewrite_lift_free_bvar_in
  invert_instantiated_tm_shape
  solve_instantiated_fvar_case

Analysis:

  These were introduced to compress the long msubst-back proof.  They are useful
  but not all have the same status:

  - rewrite_lift_free_bvar_in is a syntax rewrite around
    lstore_instantiate_value_at_lift_free_bvar.  It may be useful outside this
    proof, but its current implementation pattern-matches a concrete cbn shape
    and emits cast-in-pattern warnings.  It should not be public as-is.

  - invert_instantiated_tm_shape is proof-local.  It only consumes equalities
    generated by this mutual induction proof.  Keep local or put inside a small
    Section-local block if possible.

  - solve_instantiated_fvar_case overlaps with store/lty-env lookup automation:
    it destructs a store lookup, rewrites lstore_free_part_lift_free, then either
    reconstructs a fvar typing from atom_store_has_ltype or derives an original
    lty_env lookup from lty_env_msubst_store_lookup_some.

Recommended change:

  Add a lower helper lemma rather than exposing this Ltac:

    Lemma basic_value_has_ltype_msubst_store_back_instantiated_fvar ...

  or two smaller lemmas:

    basic_value_has_ltype_msubst_store_back_lookup_some
    basic_value_has_ltype_msubst_store_back_lookup_none

  Then solve_instantiated_fvar_case becomes a short local tactic applying those
  lemmas after the syntactic equality is inverted.

  Do not move invert_instantiated_tm_shape downward unless another file needs
  the exact same proof shape.

4. Const-specific local tactics should be folded into a const syntax normalizer
------------------------------------------------------------------------------

Current location:

  Denotation/theories/ConstDenote.v

Current Ltac:

  solve_const_guard
  solve_const_over_guard
  solve_const_under_guard
  const_scope_set
  const_forall_scope_norm
  solve_const_forall_closed_scope
  solve_const_forall_open_scope

Analysis:

  These are local and semantically const-specific, so keeping them in the const
  file is reasonable.  But they still duplicate lower syntax normalizers:

    formula_syntax_norm
    cty_lvars_syntax_norm
    cty_fv_syntax_norm
    denot_lvars_norm
    better_set_solver

Recommended change:

  Introduce one local tactic:

    Ltac const_denot_syntax_norm :=
      formula_syntax_norm;
      type_syntax_norm;
      denot_lvars_norm;
      store_normalize.

  Then define the current const tactics in terms of that normalizer and
  better_set_solver/better_store_solver.  This keeps const-specific proof
  orchestration local while reducing repeated bare rewrite steps.

5. Store rekey/map-key notation should be rationalized
------------------------------------------------------

Current files:

  ContextStore/StoreCore.v
  ContextStore/StoreInterface.v
  ContextStore/StoreFilterMapKey.v

Current notation issue:

  StoreCore has both:

    Notation "'storeA_map_key' f s" := (kmap (M2:=gmap _) f s)
    Notation "'storeA_rekey' f s" := (kmap (M2:=gmap _) f s)

  storeA_rekey is heavily used in store and lvar-store lemmas.  Earlier cleanup
  discussions considered removing rekey because it is just map-key over the same
  key type.  Current usage is still high, especially in:

    ContextStore/LVarStore.v
    ContextStore/StoreFilterMapKey.v
    ContextBasicDenotation/RelevantEnv.v

Recommendation:

  Do not delete storeA_rekey yet.  It has become the name for same-key map-key
  proofs.  But make the distinction explicit:

    - storeA_map_key: cross-key map-key, K -> K'
    - storeA_rekey: same-key map-key, K -> K

  Add a short comment in StoreCore.v explaining this semantic distinction.
  Without the comment, rekey looks like a pointless wrapper.

  Also remove duplicate storeA_map_key notation declarations if both are still
  needed only because of section placement.  Prefer one public notation near the
  first generic definition.

Notation analysis
=================

Notations that appear unused or too cute
----------------------------------------

1. ContextTypeLanguage/theories/Qualifier.v

  Notations:

    q '^q^' x
    q1 '&q' q2
    ⊤q

  Current scan:

    '&q' and '⊤q' appear only at their declarations.
    '^q^' is hard to grep because of syntax, but no obvious source use appeared
    from the direct pattern scan.

  Recommendation:

    If these are paper-facing and used interactively, keep them in Notation.v
    rather than Qualifier.v.  If not used in source and not intended for proof
    authoring, delete them.  In particular, '&q' and '⊤q' look unnecessary in
    compiled code.

2. ContextTypeLanguage/theories/TypeOpen.v

  Notation:

    η ⊙ x

  Current source use is mainly smoke tests in Notation.v.  It is visually nice
  but not widely used.

  Recommendation:

    Keep only if the paper-facing notation file wants it.  Otherwise replace
    smoke-test-only use with the underlying mopen/open_cty_env functions and
    delete the notation.

3. ContextLogic/theories/FormulaConnectives.v

  Notation:

    sat m φ := res_models m φ

  Current use:

    One definition near ext; otherwise almost unused.

  Recommendation:

    Delete sat notation or move it to FormulaNotation.v if it is desired as a
    user-facing shorthand.  In source, prefer "m ⊨ φ" or res_models.

4. ContextStore/Store.v and StoreInterface.v

  Notation:

    store_restrict := storeA_restrict
    σ |ₛ X

  store_restrict is used heavily and is clearer than storeA_restrict at call
  sites.  Keep it.

  The infix σ |ₛ X appears less used.  If it is not used outside smoke/example
  code, delete it.  If kept, use it consistently in Store-only lemmas, not mixed
  with store_restrict in the same proof.

5. Denotation/ContextBasicDenotation Notation files

  Type aliases:

    FormulaT, StoreT, LStoreT, WorldT, WfWorldT, ...

  These are heavily used and should stay as only-parsing notations.  They reduce
  noise around value-instantiated logic/resource types.

Notations worth adding or using more consistently
-------------------------------------------------

1. Denotation target notation

  ty_denote_under is conceptually a compatibility wrapper around:

    ty_denote (erase_ctx Γ) τ e

  It is used heavily in Soundness.  Do not delete until higher-order cases are
  stable.  But once those cases are proved, consider replacing it with a
  notation-only name or using ty_denote directly in theorem statements.

2. Result-equivalence package

  typed_total_equiv_on is semantically important but verbose.  If it
  keeps appearing in future proofs, add a notation such as:

    e1 ≃[Σ,τ,m] e2

  Do not add this now unless there are at least several source uses.  It could
  make proof terms harder to read if introduced prematurely.

3. Denotation model notation

  The notation "m ⊨ φ" is good and heavily used.  Keep.
