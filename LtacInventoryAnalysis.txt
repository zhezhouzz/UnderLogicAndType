Ltac Inventory and Cleanup Analysis
===================================

Scope
-----

This report scans all `.v` files for `Ltac` and `Tactic Notation`
definitions and classifies them by intent, current use, and cleanup
opportunity.  The scan is textual, so notation variants such as
`better_set_solver !` and `better_set_solver !!` are counted separately; the
counts below should be read as a guide, not as proof of semantic importance.

High-level summary
------------------

- Approximate tactic/notation definitions found: 235.
- The tactic stack is already naturally layered:
  1. pure stdpp/set/map/store solvers;
  2. locally nameless binder/freshness support;
  3. syntax/open/fv/erase/relevant-env normalizers;
  4. semantic regular extraction tactics;
  5. case-specific proof scripts for soundness, fix, persist, const, arrow/wand.
- The most-used tactics are the expected solver layer:
  - `better_set_solver`: about 448 textual hits across 41 files.
  - `my_set_solver`: about 68 hits.
  - `better_base_solver`: about 53 hits.
  - `models_fuel_irrel`: about 44 hits.
  - `better_map_solver`: about 27 hits.
- The biggest cleanup opportunities are not in the pure solver layer.  They are
  in duplicate local tactics, underused locally nameless tactics, and multiple
  overlapping open/relevant-env normalizers.

Important convention for future cleanup
---------------------------------------

Do not replace precise, deterministic normalization with a giant global solver.
The healthy shape should be:

1. run regular extraction for repo-specific hypotheses;
2. run syntax/open/relevant-env normalization at the source boundary;
3. call a small set/map/store solver only on the residual stdpp goal.

This is especially important for compile time: `better_set_solver` is powerful
but not cheap, and `fast_set_solver !!` should be preferred when it suffices.

  - normalize common stdpp set/map/base expressions;

2. Locally nameless layer

- File:
  - `LocallyNameless/Tactics.v`
- Main tactics:
  - binder/freshness:
    `collect_stales`, `auto_exists_L`, `pose_fresh_from`,
    `auto_pose_fv`, `pose_fresh_fv`, `specialize_with`
  - cofinite constructors:
    `econstructor_L`, `auto_specialization`,
    `econstructor_L_specialized`
  - binder cleanup:
    `crush_binders`, `ln_simpl`, `smart_ln_simpl`
  - variable-decide cleanup:
    `var_dec_solver`, `var_dec_simpl`, `var_dec_set_solver`
  - old apply helpers:
    `apply_eq`, `eapply_eq`, `auto_apply`, `auto_eapply`,
    `auto_apply_eq`, `auto_eapply_eq`
- Intent:
  - automate locally nameless cofinite quantification and freshness picking.
- Current state:
  - `collect_stales` is used, but not enough.  It is the right source of
    "names already in the proof state" and should replace ad hoc `L := ∅` or
    hand-built stale sets in open/fresh proofs.
  - `econstructor_L` and `econstructor_L_specialized` are underused relative
    to the number of cofinite constructors in the repo.
  - `pose_fresh_from` and `pose_fresh_fv` appear unused.  They may be old
    routes, but their idea is still useful if wrapped by a deterministic
    binder-intro tactic.
  - `auto_apply_eq` / `auto_eapply_eq` appear unused.  They are likely old
    proof-search helpers and should not be expanded further.
- Recommendations:
  - Add a small "LN opening boundary" tactic for proof scripts that:
    1. calls `collect_stales`;
    2. chooses the opened name from existing syntax names when possible;
    3. applies `econstructor_L` / `econstructor_L_specialized`.
  - Replace manual cofinite boilerplate in BasicTyping/Open/Soundness with this
    tactic, but only where timing stays stable.
  - Mark `auto_apply_eq` and `auto_eapply_eq` as removal candidates if no
    external use appears.

Underused but probably important
--------------------------------

- `collect_stales`
  - Current use is low relative to the number of cofinite/open proofs.
  - This should be the default way to build forbidden sets for binder openings.

- `auto_exists_L`
  - Should be used when a proof goal asks for a cofinite set and the current
    script currently chooses one by hand.

- `econstructor_L` / `econstructor_L_specialized`
  - These encode the intended cofinite-constructor pattern.
  - They are likely the right tool for basic typing and LN regularity proofs.

- `soundness_case_start`
  - This is the right abstraction for large fundamental-case lemmas.
  - Further shared first-step proof code should be added here only when it is
    deterministic and cheap.

- `denotation_regular`
  - Current usage is low, but the idea is right: extract repo-specific resource
    facts before handing set/map goals to stdpp.
  - It should be extended through regular lemmas, not by adding broad proof
    search.

Tactics that look like old-route leftovers
------------------------------------------

- `relevant_lvars_norm`
  - Still used, but conceptually legacy after moving toward direct support
    expressions and notation-style relevant envs.
  - Keep as compatibility for now; prefer `support_lvars_norm` in new proofs.

- `result_first_open_norm`
  - Duplicates Denotation's result-first open normalizer but is unused.
  - Either make it the public entry and replace callers, or delete it.

- `auto_apply_eq` / `auto_eapply_eq`
  - No usage and broad proof-search flavor.
  - Candidate for removal after checking no out-of-repo dependency.

- `formula_semantics_norm`
  - Could be a good public tactic, but currently unused.
  - If not adopted soon, it should be removed to avoid a misleading API.

Downshift candidates
--------------------

- Pure set/store/resource facts currently embedded in Denotation/Soundness
  local tactics should be moved downward only if they mention no Denotation or
  Soundness definitions.
- `denotation_regular_res_extend_*` may be partly resource-level, but because
  the names mention result extension, first check whether they depend on
  `res_extend_by`.  If yes, keep them in the lowest folder that defines that
  concept.

  `DenotationSetMapFacts.v`.  If they mention only stdpp and atom freshness,
  rename and move them lower; if they mention context typing definitions, move
