Change / Unfold Analysis
========================

Scope
-----

This report scans Rocq files in the current repo for explicit proof-shaping
commands:

- `change`
- `unfold`
- nearby simplification infrastructure: `class_simpl`, `cbn [...]`, and
  existing layer-specific solvers.

Raw totals from `rg`:

- `change`: 813 matches
- `unfold`: 998 matches
- `class_simpl` / explicit `cbn [...]` related matches: 641 matches

The goal is not to remove every occurrence.  The useful target is to stop
proofs from depending on ad hoc intensional shapes when a layer-specific
normalization tactic or a short semantic lemma should expose the same shape.

What the repeated `change` commands mean
----------------------------------------

1. Typeclass/notation opacity and gmap casts

   Common examples:

   - `change (dom (Σ : gmap logic_var ty))`
   - `change (((Σ : gmap logic_var ty) !! LVFree x) = None)`
   - `change (world_dom (m : WorldT) = ...)`
   - `change ((storeA_restrict σ X : gmap K V) !! x = ...)`

   These are not proof ideas.  They are shape coercions needed because notations
   like `StoreA`, `World`, `lty_env`, `store_restrict`, and `world_dom` are
   intentionally abstract-ish but many stdpp lemmas talk about plain `gmap`.

2. Store/restrict lookup and domain normalization

   Common examples:

   - `change (dom (storeA_restrict σ X : gmap K V) = ...)`
   - `change ((storeA_restrict σ X : gmap K V) !! x = ...)`
   - `unfold storeA_restrict`
   - `unfold storeA_compat, map_compat`

   Existing `better_store_solver` is the right direction.  It should be
   extended before manually rewriting upper proofs.

3. Resource/world representation exposure

   Common examples:

   - `unfold sqsubseteq, wf_worldA_sqsubseteq, resA_le, rawA_le`
   - `change (dom σ = worldA_dom w)`
   - `change (resA_restrict ... = ...)`

   This suggests a missing `resource_norm` / `resource_le_norm` tactic in
   ContextAlgebra, analogous to Store's `store_normalize`.

4. Formula semantics and scope

   Common examples:

   - `unfold res_models`
   - `unfold formula_scoped_in_world`
   - `cbn [formula_measure res_models_fuel]`

   At the Logic layer this is acceptable.  Above Logic, proofs should use
   `FormulaTactics`, `FormulaConnectivesCore`, `FormulaImpl`, `FormulaWand`,
   and `FormulaForall` APIs rather than unfolding semantics.

5. Locally nameless and syntax open/shift

   Common examples:

   - `change (cty_shift 0 τ) with (cty_shift (0 + 0) τ)`
   - `change ({d + k ~> x} CTArrow τ1 τ2) with ...`
   - `unfold lvars_open_env`
   - `unfold swap`

   Some of this belongs in definition-layer induction proofs.  In upper layers,
   this should be hidden behind syntax normalizers such as `formula_open_syntax_norm`,
   `formula_fv_syntax_norm`, and new cty/tm open normalizers.

6. Core multi-substitution and map_fold exposure

   Common examples:

   - `change (map_fold (fun x vx acc => {x := vx} acc) e σ) with (m{σ} e)`
   - `unfold closed_env`
   - `unfold lc_env`

   This is a strong target for a CoreLang `msubst_norm` tactic and possibly
   hints for `closed_env`/`lc_env` when the proof is operational/substitutional.


Recommended tactics and normalization layers
--------------------------------------------

The clean approach is not one giant `unfold_all`.  We want scoped tactics whose
names express the semantic layer they normalize.

1. Base / Store

   Add or strengthen:

   - `map_shape_norm`: normalize plain `gmap` lookup/dom shapes, singleton,
     insert, delete, union, restrict-map if still outside Store.
   - `store_shape_norm`: expose only enough of `StoreA`, `storeA_restrict`,
     `storeA_map_key`, `storeA_swap`, `storeA_shift`, `storeA_compat` to let
     map/set solvers work.
   - `store_cast_norm`: handle frequent casts like `(σ : gmap K V)` and
     notational aliases.  This should replace many one-off `change`.

   Existing `better_store_solver` is close; the main missing piece is better
   cast/notation recognition so proofs do not write `change` before calling it.

2. Algebra

   Add:

   - `resource_shape_norm`: normalize `world_dom`, `worldA_dom`,
     `resA_restrict`, `res_restrict`, `raw_world`, `lw`, `lraw_world`,
     singleton worlds, and `resA_product`/`rawA_product` only when they occur
     in map/set/domain goals.
   - `resource_le_norm`: normalize `⊑`, `sqsubseteq`,
     `wf_worldA_sqsubseteq`, `resA_le`, `rawA_le` in a controlled way.
   - `extension_shape_norm`: normalize `ext_in`, `ext_out`, `ext_rel`,
     and `mk_fiber_extension` projections.

   This should target the repeated patterns in ResourceAlgebra,
   ResourceExtension, ResourceCompat, and upper BasicDenotation extension
   proofs.

3. Logic

   Already improved:

   - Formula connective models destruct/split tactics live in ContextLogic.
   - Formula scope decomposition and forall-open scope helper live in
     ContextLogic.

   Possible additions:

   - `formula_models_norm`: controlled opening of `res_models` only at the
     Logic layer.
   - `formula_scope_norm`: currently exists partly through
     `formula_scope_syntax_norm`; it can be used more systematically in upper
     proofs.

4. TypeLanguage syntax

   Add:

   - `cty_lvars_norm`: normalize `fv_cty`, `context_ty_lvars`,
     `context_ty_lvars_at`, `context_ty_open_lvars`, and constructor lvar sets.
   - `cty_open_norm`: normalize one-step open/shift syntax without unfolding
     every recursive definition.
   - `lty_env_projection_norm`: normalize `lty_env_to_atom_env`,
     `atom_store_to_lvar_store`, and lookup/dom facts.

   These would reduce WFContextTy, TypeOpen, SyntaxLvars, LtyEnvProjection, and
   some BasicDenotation proof scripts.

5. CoreLang

   Add:

   - `msubst_norm`: rewrite `map_fold ...` back to `m{σ} e`, push
     substitution through syntax constructors, and normalize closed/lc-env
     obligations.
   - `core_sugar_norm`: normalize `tapp_tm`, `tapp_tm_fvar_spine`, and sugar
     typing shapes when the goal is syntactic or typing-related.

   This should be local to CoreLang and exported upward.

6. Denotation

   Add:

   - `denot_relevant_norm`: normalize `relevant_env`,
     `relevant_lvars`, and `lty_env_restrict_lvars` through short
     equalities/agreement lemmas.
   - `denot_open_norm`: use models-level open transport instead of formula
     equality whenever qualifier/domain paths are intensional.
   - `tapp_tlete_syntax_norm`: only for term/lvars/fv associativity facts,
     not for extracting giant denotation-shaped lemmas.

   Important guardrail: do not extract long lemmas specialized to
   `ty_denote_gas` unless the statement is genuinely a semantic API.  Many
   TLetSupport changes should become small Core syntax lemmas, Formula syntax
   lemmas, or relevant-env normalizers instead.


Which occurrences should remain explicit
----------------------------------------

Some `unfold`/`change` commands are fine and should not be hidden:

1. Definition-layer proofs in Base, Store, Algebra, Logic, TypeLanguage syntax.
   When proving the primitive lemma for a definition, unfolding that definition
   is expected.

2. Core semantics proofs where the statement is exactly about `res_models`,
   `res_models_fuel`, `rawA_le`, or `map_fold`.

3. Short proof-local changes that reveal an intentional statement shape for a
   reader, especially when the term is only changed once and is not a repeated
   pattern.

------------------------

1. Extend `better_store_solver` / `store_normalize` to absorb more gmap casts,
   `storeA_restrict` lookup/domain, and `storeA_compat` unfold patterns.
   Target files: StoreTyping.v, TermEval.v, TermExtension.v, ResourceCompat.v.

2. Add `resource_shape_norm` and `extension_shape_norm` in ContextAlgebra.
   Target files: ResourceAlgebra.v, ResourceExtension.v, ResourceCore.v,
   ResourceCompat.v.

3. Add `cty_lvars_norm` and `cty_open_norm` in ContextTypeLanguage.
   Target files: WFContextTy.v, TypeOpen.v, SyntaxLvars.v,
   TypeDenoteOpen.v.

4. Add `denot_relevant_norm` in Denotation.
   Target files: TypeDenote.v, TypeDenote.v,
   TLet.v, TLetSupport.v.

5. Add CoreLang `msubst_norm`.
   Target file: InstantiationProps.v first.


Concrete design sketch
----------------------

Use small tactics, not one global tactic:

```
Ltac store_cast_norm := ...        (* StoreA/lty_env/gmap casts *)
Ltac store_shape_norm := ...       (* restrict/dom/lookup/swap/compat *)
Ltac resource_shape_norm := ...    (* world/resource views *)
Ltac extension_shape_norm := ...   (* ext_in/ext_out/ext_rel projections *)
Ltac cty_lvars_norm := ...         (* context type lvars/fv syntax *)
Ltac cty_open_norm := ...          (* open/shift syntax *)
Ltac denot_relevant_norm := ...    (* relevant env/lvars/restrict shapes *)
Ltac msubst_norm := ...            (* map_fold substitution shapes *)
```

Then layer solvers should compose only downward:

- Denotation tactics may call TypeLanguage, Logic, Algebra, Store, Base tactics.
- BasicDenotation tactics may call CoreLang, TypeLanguage, Logic, Algebra,
  Store, Base tactics.
- Algebra tactics may call Store/Base tactics.
- Store tactics may call Base/map/set tactics.

This keeps proof scripts readable: a proof line should say the semantic layer it
is normalizing, not manually spell out a `change` into a particular `gmap`.
