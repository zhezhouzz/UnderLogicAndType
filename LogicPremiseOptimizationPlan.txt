ContextLogic premise/lemma optimization plan
============================================

Scope
-----

This pass reviews ContextLogic/theories/*.v, with the first implementation
round focused on FormulaConnectives.v and FormulaWorldExtension.v.  The goal is
to remove premises that can be derived from existing model/resource premises,
and to delete or thin lemmas that are merely aliases of stronger/general ones.

General derived facts used by the cleanup
-----------------------------------------

1. Any premise `m ⊨ p` gives `formula_scoped_in_world m p` through
   `res_models_scoped` / `res_models_fuel_scoped`.
2. If `w1 ⊑ w2`, then `world_dom w1 ⊆ world_dom w2` through `raw_le_dom`.
3. For `FAtom q`, `logic_qualifier_denote q m` already contains
   `lqual_fv q ⊆ world_dom m`, so a separate `formula_scoped_in_world m
   (FAtom q)` premise is redundant.
4. For `FStar φ ψ`, if `res_product m1 m2 Hc ⊑ m`, `m1 ⊨ φ`, and `m2 ⊨ ψ`,
   then `m` is scoped for `FStar φ ψ`; `φ` is covered by `m1`, `ψ` by `m2`,
   product dom is their union, and the product is below `m`.
5. For `FPlus φ ψ`, if `res_sum m1 m2 Hdef ⊑ m`, `m1 ⊨ φ`, and `m2 ⊨ ψ`,
   then `m` is scoped for `FPlus φ ψ`; `Hdef` identifies the two summand
   domains and the sum is below `m`.
6. A hypothesis `res_models p φ` already implies `formula_fv φ ⊆ world_dom p`;
   any lemma taking both should usually drop the explicit fv premise.

FormulaConnectives.v
--------------------

Safe first-round changes:

1. `res_models_star_intro`
   - Current redundant premise: `formula_scoped_in_world m (FStar φ ψ)`.
   - New statement should take only `res_product m1 m2 Hc ⊑ m`, `m1 ⊨ φ`,
     and `m2 ⊨ ψ`.
   - `res_models_star_iff` can drop its scope premise and use the new intro.

2. `res_models_plus_intro`
   - Current redundant premise: `formula_scoped_in_world m (FPlus φ ψ)`.
   - New statement should take only `res_sum m1 m2 Hdef ⊑ m`, `m1 ⊨ φ`,
     and `m2 ⊨ ψ`.
   - `res_models_plus_iff` can drop its scope premise and use the new intro.
   - `res_models_plus_intro_from_models` becomes a thin alias and can either
     be deleted or left as compatibility wrapper.  In this round, keep it as a
     one-line wrapper if upper layers still mention it.

3. `res_models_atom_intro`
   - Current redundant premise: `formula_scoped_in_world m (FAtom q)`.
   - New statement should take only `logic_qualifier_denote q m`.

4. `res_models_resource_atom_intro`
   - Current redundant premise: `formula_scoped_in_world m (FResourceAtom D P)`.
   - New statement should take only `logic_qualifier_denote (lqual D P) m`.

5. `res_models_resource_atom_witness_intro`
   - Current redundant premises: both resource-atom scoped premises.
   - New statement should take only witness denotation on `m0` plus `m0 ⊑ m`.
   - The proof uses `logic_qualifier_denote_mono`; its explicit `lqual_fv`
     premise is obtained from the witness denotation.

6. `res_models_plus_map`
   - Current scope premise can be derived after transforming the two component
     models and rebuilding `FPlus`; replace the proof through the scope-free
     `res_models_plus_iff`/intro.

Potential but not first-round changes:

7. `res_models_or_intro_l_from_model` / `_r_from_model`
   - The missing branch still needs an fv/scope premise because a model of one
     side does not scope the other side.  Keep as-is.

8. `res_models_over_intro_same` and `res_models_under_intro_same`
   - The scope premise is redundant when there is `m ⊨ φ`, and the file already
     has `_from_model` versions.  Later we can either swap names or delete the
     older scoped versions after call-site review.

9. `res_models_FFibVars_intro` / projection intro / iff
   - Do NOT remove scope premise in the current semantics without an additional
     premise such as `lvars_fv D ⊆ world_dom m`.
   - Reason: if `D` contains an unused atom outside `m` and `φ = FTrue`, the
     fiber premise can still hold for the projection on `D ∩ world_dom m`, but
     `m` is not scoped for `FFibVars D FTrue`.
   - So the proposed removal is not valid unless Algebra fiber projection is
     strengthened to require the requested input domain to be present.

10. Impl lemmas
   - Keep the future/local distinction for now.  `res_models_impl_intro`,
     `res_models_impl_iff`, and `res_models_impl_elim` are used heavily by
     Denotation/TLet.  `res_models_impl_map` and `_future` are both meaningful
     because the future variant is the Kripke-correct general form.
   - A later deletion pass can remove unused aliases only after call-site scan.

11. Forall lemmas
   - `res_models_forall_map` is a pure alias for
     `res_models_forall_map_same_fv`; no call site currently needs the alias.
   - `res_models_forall_congr_same_fv` is a derived two-way map; keep if used
     by future scripts, otherwise it can be regenerated from two map calls.
   - `res_models_forall_map_subset_fv`, `res_models_forall_transport`,
     `res_models_forall_rev`, `res_models_forall_rev_intro`,
     `res_models_forall_full_world_map`, and `res_models_forall_ext_transport`
     are genuinely different interfaces and should stay.

FormulaWorldExtension.v
-----------------------

Safe first-round changes:

1. `res_models_pullback_subset_projection`
   - Drop premise `formula_fv φ ⊆ world_dom p`; it follows from `res_models p φ`.

2. `res_models_plus_extend_pullback_agree_on`
   - After changing `res_models_pullback_subset_projection`, the two local
     proofs of `Hn1f` and `Hn2f` simplify.

Careful / not safe in this round:

3. `res_models_plus_extend_pullback`
   - The explicit premise `fiber_extension_functional_on m F` is required by
     `res_extend_by_sum_pullback`.
   - I do not see an existing Algebra lemma proving `res_extend_by m F n ->
     fiber_extension_functional_on m F`.  Under the current relational
     extension definition this implication is not generally valid: an extension
     may relate the same input projection to multiple output stores, and
     `res_extend_by` then includes all of them in `n`.
   - Therefore this premise should stay unless Algebra changes extension
     semantics to enforce functionality.

4. `res_models_plus_extend_pullback`
   - The premises `world_dom m ⊆ formula_fv φ1/φ2` are a strong way to get the
     two component domains needed by the Algebra pullback.  The agree-on version
     below already avoids them by projecting the plus witnesses first; prefer
     call sites to use `res_models_plus_extend_pullback_agree_on`.
   - Later deletion candidate: remove the older strong-domain version if no
     upper file uses it.

5. `res_models_forall_map_same_fv_by_extension` and
   `res_models_forall_transport_by_extension`
   - These are thin wrappers around FormulaConnectives forall map/transport.
     No call site currently mentions them.  Candidate for deletion after the
     first proof-preserving cleanup.

FormulaSemantics.v
------------------

1. Scope decomposition lemmas are not redundant: they feed formula_scope_step
   and upper scope solvers.
2. `res_models_and_intro`, `res_models_or_intro_l/r` still require explicit
   scope because the raw semantics has scope at every formula node.  The
   higher-level convenience lemmas in FormulaConnectives should hide those
   premises where models determine scope.
3. `res_models_impl_refl` still needs `formula_scoped_in_world m φ`; there is
   no model premise from which to derive it.

FormulaSyntax.v / LogicQualifier.v / Notation / Interface
---------------------------------------------------------

1. FormulaSyntax lemmas are structural and should stay; they are now feeding
   formula syntax normalization tactics.
2. LogicQualifier denotation lemmas are already minimal enough for this pass.
   A future pass could simplify `logic_qualifier_denote_mono` by deriving the
   explicit `lqual_fv q ⊆ world_dom m0` premise from `logic_qualifier_denote q
   m0`, but it is small and local.
3. FormulaNotation.v and LogicInterface.v do not contain substantial proof
   lemmas to optimize in this pass.
