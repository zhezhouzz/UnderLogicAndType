# Proof Pattern Refactor Analysis

Date: 2026-06-14
Branch: codex/result-first-arrow-wand

## Goal

This report scans Denotation and Soundness proofs for long proof scripts,
especially those repeatedly handling the result-first denotation shape:

  forall f. expr_result_formula_at ... e f -> ... (ret f)

and the recurring normalization/set/map/store side conditions around it.

The purpose is not to change proofs in this pass, but to identify reusable
semantic lemmas and folder-local set/map tactics that should be extracted next.

## Method

Files scanned:

- Denotation/theories/*.v
- ContextTyping/theories/Soundness*.v

Long proofs are lemmas/theorems over 100 lines.  For each long proof I counted
rough proof-shape indicators:

- assert/pose proof/set/replace/change
- formula_open, expr_result_formula_at, FForall/FImpl
- typed_lty_env_bind, relevant_env
- set_solver/better_set_solver
- map/store/resource operations such as lookup_insert, map_lookup, storeA_,
  res_restrict, res_extend_by, world_dom

There are 67 lemmas over 100 lines in the scanned files.

## Repeated Result-First Forall/Impl Pattern

### Shape

Many proofs manually do this sequence:

1. Obtain or introduce an outer forall:

     res_models_forall_open_named_fresh
     res_models_forall_rev_intro

2. Open:

     formula_open 0 f
       (FImpl (expr_result_formula_at ...) (arrow/wand_value_denote ...))

3. Normalize:

     cbn [formula_open]
     formula_open_expr_result_formula_at_shift0
     formula_open_result_first_fun_arg_two
     formula_open_result_first_fun_result_two
     typed_lty_env_bind_open_current

4. Eliminate implication:

     res_models_impl_elim

5. Move from outer result graph to value-level denotation:

     arrow_value_denote_gas_with ...
     wand_value_denote_gas_with ...

Denotation:

### Extraction candidates

1. Denotation-level outer result eliminator:

   result_first_open_result_elim :
     m |= FForall (FImpl (result_graph e) value_body) ->
     fresh f ->
     mf extends m with f ->
     mf |= opened result_graph ->
     mf |= opened value_body

   This should encapsulate:

   - res_models_forall_open_named_fresh
   - formula_open normalization for expr_result_formula_at
   - res_models_impl_elim

2. Result-first value open normalization:

   arrow_value_open_two_normalized :
     formula_open 0 y (formula_open 1 f (arrow_value_denote ...)) =
     normalized ty_denote_gas with two inserted free variables

   wand_value_open_two_normalized :
     analogous for FBWand

   Existing lemmas:

   - formula_open_result_first_fun_arg_two
   - formula_open_result_first_fun_result_two

   They should be wrapped into higher-level normalize lemmas that hide the
   repeated typed_lty_env_bind_open_current side conditions.

3. Result-first environment agreement:

   result_first_value_env_agree :
     if relevant_envs agree on context_ty_lvars of arrow/wand value body,
     then opened arrow/wand value denotation transports across envs.

   Current repeated local facts:

   - arrow_result_first_result_env_agree
   - wand_result_first_result_env_agree
   - appd_open_result_env_agree
   - many hand-written relevant_env subset assertions.

## Immediate Refactor Priority

1. Add ResultFirstOpen.v and extract the outer forall/impl opening pattern.
   This directly attacks the largest duplicated shape and should reduce
   TypeEquivArrowResultFirst, TypeEquivWandResultFirst, SoundnessLam,
   SoundnessApp, SoundnessAppD, and SoundnessFixApply.

2. Add DenotationSetMapFacts.v for store/restrict/singleton extension facts.
   This reduces TypeEquivTerm and TypeEquivFiberBase, which contain most of the
   fragile set/map/store proof code.

3. Add SoundnessSetMapFacts.v with fresh/domain solvers and small wrappers.
   This reduces SoundnessLam/App/AppD/FixApply without polluting Denotation.

4. Move SoundnessMatch branch-sum result split support down to Denotation if it
   does not depend on typing.  The current Match file still contains
   denotation-level CTSum proof support.

5. Split fix_self_rec_call_reducible_measure_step into semantic phases only
   after the shared set/fresh helpers exist; otherwise the split will just move
   noise around.

## Notes On What Not To Do

- Do not replace semantic steps by a giant solver.  The repeated proof
  structure is semantic: result graph opening, extension construction, and
  environment transport.

- Do not revive old deterministic alias routes.  Current nondeterministic-ready
  result-first semantics makes TLet/result graph the right boundary.

- Do not inline TLet admits into TypeEquiv.v.  Keeping result alias delegated to
  TypeEquivTLet is the right organization until the TLet Arrow/Wand obligations
  are fully proved.

- Avoid putting Soundness-specific ctx/typing lemmas in Denotation.  Denotation
  should receive generic lty_env/store/resource facts; Soundness can wrap them
  with context_typing_wf premises.
