# Persistence And Fiberwise Joinability Optimization Plan

## Purpose

This optimization pass focuses on the new persistence layer:

- `FPersist`
- `CTPersist`
- `persistent_formula`
- `fiberwise_joinable_on`
- persistent intro for values
- persistent-over parameter reasoning

The goal is to make the new persistence proofs reusable and reviewable without
deleting the lemmas that were introduced during the first proof pass.

## Skill Guidance Used

- Build semantic helper lemmas with short statements.
- Keep fiber observation domains distinct from resource subset/projection.
- Normalize proof boundaries before extracting helpers.
- Lower Logic-level facts from Denotation/Soundness into Logic/Algebra when
  possible.

## Concrete Optimization Work

1. Split persistence proof layers.
   - Logic persistence laws remain in Logic.
   - Type-level `CTPersist` facts remain in Denotation.
   - Typing rule soundness remains in ContextTyping.

2. Normalize type-level persistence statements.
   - Use `ty_denote` layer when gas differences are irrelevant.
   - Use normalized body abbreviations for `CTOver`/`CTUnder` result bodies.
   - Avoid public statements with raw `formula_open` or `lvars_open`.

3. Strengthen reusable singleton/fiber lemmas.
   - ret-value result slot singleton construction;
   - singleton projection implies `FPersist`;
   - fiberwise aggregation through `over_open_body` and `under_open_body`;
   - opened world/fiber commute for `FForall`.

4. Make persistent intro proof shorter.
   - Move ret-value result extension facts down.
   - Move context observation coverage facts into SoundnessSetMapFacts if they
     require typing, otherwise lower them.
   - Keep `fundamental_persist_intro_case` as a short orchestration lemma.

5. Preserve all persistence lemmas.
   - Do not delete failed or narrow lemmas during this pass.
   - If a lemma is too specific, keep it as a wrapper over a more general fact.
   - Add comments explaining why arbitrary Arrow result generalization is not
     claimed.

## Current Conceptual Boundaries

- `FPersist P` is Logic-level singleton projection persistence.
- `CTPersist τ` is Type-level result-first persistence.
- Value intro is allowed only for `tret v`, not arbitrary terms.
- General Arrow reverse for persistent over parameters is not claimed; Wand and
  joinable-result facts should be handled separately.
- Dependent precise-context duplication remains out of this optimization pass.

## Implementation Status

- [x] Began shortening the value-only persistent intro support by lowering the
  value-specific relevant-env normalization out of `SoundnessPersist.v`.
  `TypePersist.vo` currently compiles but takes about 75s, so later
  compile-time work should split or optimize that file before more large
  persistence proofs are added.
- [x] Split type-level persistence support by proof role.  The public
  `TypePersist.v` import remains stable, while base facts, Arrow forward facts,
  singleton/result-slot facts, and Wand equivalences now live in separate files.
  This makes future persistence optimization target `TypePersistWand.v`
  directly instead of recompiling unrelated base facts.
- [x] Split Wand persistence again into forward and reverse files.  The reverse
  file contains the hard joinable-result direction and is now isolated for any
  future proof simplification or `-time` pass.
