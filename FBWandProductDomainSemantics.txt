# FBWand Product-Domain Semantics

## Summary

`FBWand d A B` is a binder-aware resource implication.  It should be read as
"for every argument resource that composes with the current closure resource,
and whose product introduces exactly the `d` freshly opened variables, `A`
entails `B` on the product world."

This connective is not the old ordinary `FWand`; it is the connective needed by
locally nameless context-type denotation.  The bound variables introduced by
`FBWand` are provided by the argument/product side, not by an outer `FForall`
extension of the closure world.

## Intended Semantics

The semantic clause should have the following shape:

```coq
m |= FBWand d A B
iff
exists L,
  forall η n Hc,
    open_env_binds d η ->
    open_env_atoms η ## L ->
    world_dom (res_product n m Hc : WorldT) =
      world_dom (m : WorldT) ∪ open_env_atoms η ->
    n |= formula_open_env η A ->
    res_product n m Hc |= formula_open_env η B.
```

Important points:

- `Hc : world_compat n m` must remain an explicit premise.  The argument
  resource is still composed with the closure resource using the ordinary
  resource product.
- We do **not** require `res_restrict n (world_dom m) = m`.  That would be much
  stronger than compatibility and would incorrectly force the argument resource
  to be a closure extension.
- We do **not** restrict `world_dom n` directly.  The restriction is on the
  product world:

  ```coq
  world_dom (res_product n m Hc) =
    world_dom m ∪ open_env_atoms η
  ```

  This says the whole application/product introduces exactly the `d` bound
  variables opened by `η`, and no unrelated fresh variables.
- This definition keeps the BI-style implication shape:

  ```coq
  n |= A  ->  n * m |= B
  ```

  while making the binder/FV behavior compatible with locally nameless syntax.

## Why This Fixes The Earlier Problem

The old route tried to make ordinary wand work with formula minimality:

```coq
m |= P -> res_restrict m (formula_fv P) |= P
```

For unrestricted ordinary wand this can fail, because the restricted world can
be compatible with argument resources that carry unrelated variables not
compatible with the original full world.

The new `FBWand` avoids this without replacing resource compatibility:
an argument/product world is only considered when the product domain is exactly
the closure domain plus the opened binder atoms.  Therefore unrelated variables
cannot be introduced merely to satisfy the argument formula.

## Reviewer-Facing Well-Formedness

The new semantics is intentionally domain-restricted.  A reviewer may ask
whether it is still BI-like.  The answer should be recorded by a small
well-formedness relation and a theorem showing that, on well-formed formulas,
`FBWand` entails the ordinary BI-style implication behavior.

Define a lightweight syntactic predicate:

```coq
Fixpoint formula_wf (φ : FormulaT) : Prop :=
  match φ with
  | FTrue | FFalse | FAtom _ => True
  | FAnd A B | FOr A B | FImpl A B
  | FStar A B | FPlus A B =>
      formula_wf A /\ formula_wf B
  | FBWand d A B =>
      formula_wf A /\
      formula_wf B /\
      lvars_fv (formula_lvars_at d A) ⊆
        lvars_fv (formula_lvars_at 0 B)
  | FForall A | FOver A | FUnder A => formula_wf A
  | FFibVars _ A => formula_wf A
  end.
```

This `formula_wf` should **not** be threaded through the main semantics,
minimality lemmas, denotation definitions, or Fundamental proof.  It exists only
as reviewer-facing metatheory.

The `FBWand` case says:

```coq
lvars_fv (formula_lvars_at d A) ⊆
  lvars_fv (formula_lvars_at 0 B)
```

Intuition: after the `d` bound variables supplied by the wand argument are
accounted for, the argument formula `A` cannot mention any closure/resource
variable that the result formula `B` does not mention.  Thus `d` fully accounts
for the new variables introduced by the argument side.

## Reviewer-Facing BI Theorem

The theorem should be named and commented so cleanup/refactor passes do not
delete it as an unused lemma.  Suggested names:

```coq
res_models_fbwand_wf_bi
res_models_fbwand_bi_of_wf
```

Suggested theorem shape:

```coq
Lemma res_models_fbwand_bi_of_wf m d A B :
  formula_wf (FBWand d A B) ->
  m |= FBWand d A B ->
  exists L,
    forall η n Hc,
      open_env_binds d η ->
      open_env_atoms η ## L ->
      n |= formula_open_env η A ->
      res_product n m Hc |= formula_open_env η B.
```

The proof idea is:

1. Given arbitrary compatible `n`, restrict it to the variables relevant to the
   opened argument/product:

   ```coq
   n0 := res_restrict n
     (world_dom m ∪ open_env_atoms η)
   ```

2. Show the restricted product satisfies the product-domain side condition:

   ```coq
   world_dom (res_product n0 m Hc0) =
     world_dom m ∪ open_env_atoms η
   ```

3. Use `formula_wf` plus scopedness of `B` under the product to show
   `formula_open_env η A` does not depend on any variables removed from `n`.

4. Transport:

   ```coq
   n |= openη A
   ->
   n0 |= openη A
   ->
   n0 * m |= openη B
   ->
   n * m |= openη B
   ```

   The last step uses resource monotonicity/Kripke monotonicity.

This theorem is a sanity/communication result: it explains that the
product-domain `FBWand` is BI-compatible for formulas whose argument side does
not mention variables outside the result side except for the binders introduced
by `FBWand`.

## Important Cleanup Note

Do not delete `formula_wf` or `res_models_fbwand_bi_of_wf` merely because they
are not used by `ty_denote_gas`, minimality, or Fundamental.  They are intended
as reviewer-facing metatheory explaining why the binder-aware semantics remains
BI-like on well-formed formulas.
