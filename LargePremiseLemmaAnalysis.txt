Large-premise lemma analysis
============================

Scope
-----

This report scans the current direct-Soundness branch for lemmas whose
statements carry large, shape-specific premise bundles.  The goal is not to
delete useful semantic boundaries.  The goal is to identify helpers that are
too specialized to a concrete Denotation shape, are used only once, or encode a
generic Logic connective pattern directly in a Denotation statement.

Search method
-------------

Candidates were selected by statement size and premise count:

  - statement length >= 8 lines;
  - or many [forall]/[->] premises;
  - or many concrete Logic constructors such as [FAnd], [FImpl],
    [FForall], [FWand], [FPlus], [FStar].

Each candidate was then checked with full-repo [rg] to distinguish real reuse
from definition-only artifacts.

Per-file findings
=================

Denotation/theories/TypeDenoteOpen.v
------------------------------------------------

Large candidates:

  - [res_models_open_arrow_body_iff]
  - [res_models_open_wand_body_iff]
  - [res_models_open_denot_ty_lvar_guard_to_open]
  - [res_models_open_denot_ty_lvar_guard_iff]
  - [res_models_open_ty_denote_gas_iff]
  - [formula_open_over_body]
  - [formula_open_under_body]

Assessment:

  These are the cleanest immediate cleanup target.  They are all definition-only
  artifacts in the current repo.  The Arrow/Wand lemmas also have the exact bad
  shape we want to avoid: a large Denotation-specific statement spelling out a
  [FForall] body with nested [FImpl]/[FWand], plus a recursive-open hypothesis.
  Current callers use the simpler equality
  [formula_open_ty_denote_gas_singleton] directly.

Action:

  Delete the unused lemmas and the old proof-plan comment above them.  Keep
  only:

    - [ty_denote]
    - [formula_open_ty_denote_gas_singleton]

  If an opened-body transport is needed again, rebuild it as a Logic-level
  connective lemma, e.g. a generic [FForall + FImpl/FWand] map, and keep the
  Denotation side as a short normalization wrapper.

Denotation/theories/TypeEquivArrow.v
---------------------------------------------------------

Large candidates:

  - [ty_denote_gas_tm_result_equiv_arrow_open_result]
  - [ty_denote_gas_tm_result_equiv_arrow_body]
  - [basic_world_formula_arrow_open_result_target]
  - [expr_basic_typing_formula_arrow_open_result_target]
  - [typed_total_equiv_arrow_open_result_mid]
  - [basic_world_formula_arrow_open_result_big]
  - [basic_world_formula_arrow_open_result_subenv]
  - [ty_denote_gas_tm_result_equiv_arrow_open_arg]

Assessment:

  These helpers are used by the result-equivalence proof.  Most are used once,
  but inlining them would make the Arrow proof less readable and would move the
  large proof obligations back into the main induction.  The real common
  pattern is not an inline candidate; it is the package:

    opened body source/mid/target env-agree
    + target zero guard
    + recursive result-equivalence package

  This should eventually become a smaller opened-result package lemma shared
  with Wand.  That is a semantic proof refactor, not a safe first cleanup.

Action:

  Keep for now.  Do not inline.  Future cleanup should introduce a shared
  Arrow/Wand opened-result package with short premises, after the higher-order
  soundness route is revisited.

  Same as Arrow, with extra resource/product-world obligations.  These are

  resource-specific product-world transport helper.

Action:

  Keep for now.  Future cleanup should share the package with Arrow where the
  statements can remain short.

Denotation/theories/TypeDenote.v
------------------------------------------------------

Large candidates:

  - [formula_open_env_denot_guard]
  - [ty_denote_gas_scope_of_guard]
  - [denot_guard_term_type_fv_scope]

Assessment:

  [formula_open_env_denot_guard] is reused several times inside the structural
  open proof and is a legitimate boundary.  Its long statement comes from the
  guard being spelled out as nested [FAnd].  The right cleanup is a definition:

    ty_guard_formula Σ τ e

  followed by tiny syntax lemmas for opening/fv/models.  That change touches
  the core definition of [ty_denote_gas] and many proofs, so it should be a
  separate semantic-neutral refactor.

Action:

  No code change in this pass.  Recommended future pass: introduce
  [ty_guard_formula] and replace repeated nested [FAnd] statements.

  transport packages are introduced, revisit Over/Under.

  - [ctx_denote_under_comma_bind_from_result_extension]
  - [fundamental_let_case]

Assessment:

  They should shrink after [ty_guard_formula].  [fundamental_let_case] is a

  lemmas again.

Action:

  No code change in this pass.
